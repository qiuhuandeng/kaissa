# 第 5.1 步：单团自组全链路验收报告

## 验收范围

本轮只验收“产品中心 > 自营产品 > 单团项目”从项目创建到业财承接的主链路，不做真实跨页数据回写，不新增动态模拟。

主链路：

产品中心 > 自营产品 > 单团项目 → 单团项目详情 → 确认成团 → 销售中心 > 订单管理 > 项目订单详情 → 销售中心 > 合同管理 → 履约中心 > 团期管控/项目团详情 → 履约中心 > 团期结算 → 销售中心 > 收款认领 / 财务 > 应收管理 / 财务 > 收款认款 → 财务 > 业务结算 → 财务 > NC推送。

说明：当前实际导航命名为“销售中心”“会员”，全局业务口径里常写“销售中心”“客户”。本报告按当前页面可达菜单描述，同时保留业务含义。

## 验收方式

- 已完成源码级链路核对：页面入口、参数、抽屉内容、状态语义、财务承接方向。
- 已复用第 4 步硬检查结果：HTML/共享脚本可解析，静态 HTML href 无断点，导航 pageOwner 无缺失。
- 未完成人工浏览器点击：本轮 in-app browser 不可用，因此不能把本报告表述为“人工点击已通过”。

## 总体结论

源码级判断：单团自组主链路可跑通，未发现 P0 链路断点。

关键业务口径基本成立：

- 单团项目不从普通订单录入入口创建项目订单，而是在“确认成团”后生成项目订单、项目合同、项目团、阶段应收计划。
- 项目订单的收款动作已经使用“登记阶段收款”口径，不是散拼团费收款。
- 项目团已经区别于公开散拼团期，列表和详情弱化公开渠道余量、OTA、可售库存语义。
- 成本、回团结算依据、付款申请集中在“履约中心 > 团期结算”承接。
- 财务只处理应收、认款、结算、NC，不回头维护名单和资源。

## 发现的问题

### P1-001 项目团编号口径不一致

同一个单团项目在上游和财务侧出现两套项目团编号：

- 单团项目、项目订单、合同、应收、项目团详情使用：`MICE20250915001`
- 团期成本、财务 > 业务结算、NC 使用：`PRJ-MICE-20260915-001`

这不是静态链接断点，但会影响业务人员和财务核对同一项目团。建议下一轮统一编号口径，优先以上游“确认成团”生成的项目团编号为准；如果确实需要 `PRJ-MICE-*`，应明确它是“项目结算辅助核算号”，并在所有相关页同时展示映射关系。

### P2-001 菜单命名与业务目标存在差异

当前实际导航里是“销售中心 > 订单管理/收款认领/合同管理”，全局业务口径里写的是“销售中心”。这不会导致链路断点，但最终交付前需要统一命名口径，否则培训和验收报告容易混乱。

### P2-002 回团结算入口收敛在团期成本内

当前没有单独的 `回团结算.html` 页面；回团结算依据、客户验收单、提交项目结算都在 `履约中心 > 团期结算` 的结算依据抽屉中承接。这符合“所有团都在团期成本完成”的收敛原则，但如果最终菜单仍要求“产品中心 > 自营产品 > 回团结算”，需要决定是恢复菜单入口指向同一页面的 `tab=settlement`，还是把菜单规范改为只保留“团期结算”。

## 本轮不判定为问题的点

- 单团项目详情里有“NC推送”反查入口：当前只是跳到财务 NC 页面，不是在业务页直接推送，因此不算业财越界。
- 项目订单里有“结算”入口：当前文案明确实际成本、应付、付款和 NC 在团期成本、回团结算和财务模块处理，因此只是承接入口。
- `booking.html` 能识别 `customProject` 来源：只作为来源承接和防误入，不作为单团主下单入口。
