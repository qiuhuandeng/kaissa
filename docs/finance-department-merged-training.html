财务部门系统培训融合讲稿

融合原则：以“主链版”为骨架，以“能力补充版”为场景细节。主线不按菜单讲，而按财务验收业务数据的三条链讲：收入链路、成本付款链路、结算链路。发票、多币种、银联水单、预收预存预付、月结、报表、NC 等能力不单独打散，而放回对应场景里讲。

一、融合判断：两份材料各自承担什么

材料优势融合处理
Untitled (1).docx结构更聚焦，围绕财务定位、三条主链、八个场景、数据质量、日常工作和异常边界展开，适合做培训主稿。作为合并稿主骨架，保留讲课顺序和财务验收口径。
Untitled.docx细节更丰富，补充了银联/水单、自动开票、多币种汇率、资金池、期间关账、报表、预付预存等财务关心点。拆成场景补充，嵌入收入、成本付款、结算、NC 和报表关账环节，不单独作为主结构。

对财务的核心表达：这套 ERP 不是让财务多录一遍业务数据，而是让业务数据在源头结构化。财务从人工录入、人工追数，转向核对、例外处理、规则管理、入账和对外支付。

二、财务培训的总定位

财务部门这场培训不能讲成“财务模块功能介绍”，而要讲成：这套 ERP 怎么把业务数据自动变成财务可核对、可结算、可入账、可推 NC、可出报表的数据链路。

财务不是单纯使用者，而是整个业务链路的最终验收者。销售、产品、计调、履约完成业务动作后，系统最终要让财务能回答：

  • 这笔钱该不该收，收多少，谁来付；
  • 钱有没有到账，到账后认到哪张订单、哪张应收；
  • 这笔成本有没有业务依据，能不能形成应付；
  • 付款申请是否超额，审批通过后是否已经实际付款；
  • 发票、回单、汇率、结算、NC 是否能对上；
  • 报表里的每个数字能不能穿透到源单。

财务培训要讲清七件事

问题讲法
哪些动作自动形成财务数据订单、收款、退款、成本确认、结算申请、付款申请、发票处理如何触发财务对象。
财务数据从哪里来订单、合同、团期、项目、服务单、到账流水、供应商账单、发票、回单。
财务需要核什么金额、对象、主体、币种、汇率、附件、票据、账户、NC 口径。
哪些能自动处理自动生成应收、自动识别流水、自动认款、自动生成开票任务、自动生成 NC 任务。
哪些异常必须介入认款不匹配、成本依据缺失、供应商账单差异、账户异常、发票不一致、NC 推送失败。
上游数据不准怎么办财务通过退回、卡控、异常提示倒逼业务补齐源头数据。
最终如何闭环结算、付款、发票、回单、NC 凭证和报表如何形成完整链路。

三、业财一体化总链路

产品/线路/团期/项目/服务单 -> 销售订单/合同/售后 -> 应收/到账流水/认款/预收/退款 -> 履约成本/供应商账单/应付 -> 付款申请/审批/付款执行/回单 -> 业务结算/门店结算/渠道结算 -> 发票/NC/报表/关账

四、财务核心对象

对象含义来源去向
应收客户、门店、渠道或项目应该付给公司的金额。订单、合同、项目阶段款、补差、应收调整。收款认款、发票、结算、NC。
到账流水银行、银联、POS、支付平台或上传水单形成的资金流水。银行接口、银联水单、POS、微信支付宝、门店上传、Excel 导入。流水识别、认款、预收、预存、退款退回。

预存门店、渠道或客户账户余额。门店充值、渠道预存、账户扣款、退款退回。订单扣款、门店对账、资金池、NC。
退款已审批的退款执行对象。售后退款、应收冲减、预存退款。退款执行、银行回单、发票红冲、NC。
供应商账单供应商提供的成本账单。供应商上传、邮件导入、接口、履约成本。应付确认、进项发票、结算。
应付已确认的付款义务。成本确认、供应商账单、采购批次、服务单代垫。付款申请、付款执行、发票、结算、NC。
付款申请基于应付或预付节点发起的付款申请。业务侧成本页、应付单、合同节点、供应商账单。审批、付款执行。
付款/回单出纳实际付款结果和银行证明。已审批付款申请、银行回单、银企直联。应付状态、回单核销、NC。
结算对团期、项目、订单、门店、渠道的财务确认。收入、成本、退款、应付、付款、发票、回单。报表、NC、关账。
发票销项开票、进项收票、红冲、验真等税务对象。应收、收款、应付、付款、退款、结算。税务台账、NC、结算风险。

必须讲清：到账流水不是收款凭证；付款申请不是付款结果；结算确认不是业务执行。财务要按对象和状态处理，不能把不同环节合成一个动作。

五、三条主链讲法

第一条：收入链路

订单 -> 合同/应收 -> 到账流水 -> 收款认款 -> 应收核销 -> 发票 -> 收入确认/NC
  • 订单确认后，系统自动形成应收。
  • 客户付款后，银行、银联、POS、门店水单进入到账流水。
  • 系统按金额、付款方、订单号、附言自动匹配，财务处理未匹配和差异。
  • 认款确认后核销应收，生成收款凭证和 NC 任务。
  • 超收进入预收，门店/渠道充值进入预存账户。
  • 发票根据应收、收款和客户抬头自动形成任务。

第二条：成本付款链路

团期/项目/服务成本 -> 应付确认 -> 付款申请 -> 审批 -> 付款执行 -> 回单 -> 发票 -> NC
  • 团期回团、项目完成、服务单完成后，成本进入财务复核。
  • 供应商账单和业务成本匹配后，形成应付。
  • 预付不是成本，必须后续冲抵实际应付。
  • 付款申请要区分应付金额、可申请金额、本次申请金额。
  • 审批通过不等于已付款，出纳执行并回单核销后才完成。
  • 进项发票、回单和 NC 是付款链路的证据闭环。

第三条：结算链路

团期结算/项目结算/渠道结算/门店结算 -> 财务复核 -> 差异处理 -> 结算确认 -> NC/报表
  • 财务复核收入、成本、退款、应付、付款、发票、回单和毛利。
  • 门店月结核订单、收款、水单、预存、退款和差异。
  • 渠道/OTA 对账核平台账单、订单、佣金扣点、退款和应回款。
  • 结算确认后进入报表和 NC。
  • 已结算后补账单、退款、汇率差异、发票红冲必须走结算调整。

六、场景一：订单产生应收

业务背景：销售订单确认后，系统不应该等财务重新录一张应收。订单金额、客户、合同、产品、团期、渠道、收款计划明确时，系统应自动形成财务应收。

财务讲解顺序

  1. 销售创建订单，客户、产品、团期、合同、金额和付款节点完整。
  2. 系统自动生成应收单和应收计划。
  3. 财务进入应收管理，按订单号、客户、渠道、团期或项目筛选。
  4. 核对应收来源、应收金额、已认金额、待收金额、到期日和账龄。
  5. 如果来源完整，确认应收或等待认款；如果金额、主体或客户信息异常，退回业务源单。
财务要核什么判断口径
来源订单订单是否真实有效，合同和客户信息是否完整。
金额应收金额是否等于订单财务口径，优惠、补差、退差是否已审批。
主体财务主体、门店、渠道是否准确。
币种汇率外币业务是否保留原币、本币、汇率和汇率来源。
状态待收、部分收款、已收清、逾期是否准确。

七、场景二：客户付款到账与认款核销

业务背景：客户付款到账后，系统要先形成到账流水，再通过自动识别或人工认款核销应收。到账不等于收款完成，认款确认后才算财务核销。

到账来源系统处理方式
银行流水银行接口同步或 Excel/CSV 导入。
银联/POS接口同步或上传银联水单、POS 清算文件。
微信/支付宝支付平台自动同步。
门店收款门店上传水单或总部导入。
OTA/渠道回款平台账单、渠道对账单、回款流水匹配。
线下特殊收款财务登记流水，但仍需识别和认款。

演示流程

  1. 进入到账流水，演示银行流水导入或银联水单上传。
  2. 查看导入预览：流水号、到账时间、付款方、金额、重复校验。
  3. 确认导入后，流水进入待识别或已识别状态。
  4. 从已识别流水进入收款认款。
  5. 查看系统匹配的应收单和订单。
  6. 一笔流水对应多张订单时，演示拆分认款；多笔流水对应一张订单时，演示分次核销。
  7. 到账金额大于应收时，差额转预收。
  8. 确认认款后，应收状态更新，生成收款凭证和 NC 任务。

财务视角重点：财务不再逐笔翻订单和流水，而是处理系统无法确定的异常，例如付款方匹配多个客户、金额存在拆分差异、银联/POS 清算差异、认错款冲正。

八、场景三：预收、预存、转款与退款反冲

收入链路里容易被讲散的就是余额和售后。融合时要把它们放在认款之后讲，因为它们本质都是“钱已发生，但还没有简单核销到一张应收”的处理。

对象场景财务处理
预收客户超收、企业预付款、退款转预收。形成客户或订单层余额，后续抵扣新订单、退款或冻结。
预存门店充值、渠道预存、账户扣款。形成账户余额台账，进入门店对账和订单扣款。
转款核销转团、改期、改舱、改铺。核对原订单已收、新订单应收、可转金额和差额去向。
退款反冲售后退款、退差。未收部分冲应收，已收部分形成退款单，已开票触发红冲。

必须强调：售后审批通过不等于已经退款。财务执行退款并核销回单后，退款才完成。财务不处理客户为什么退，也不修改库存释放规则；财务处理可退金额、退款账户、付款执行、回单、发票红冲和凭证。

九、场景四：团期/项目成本形成应付

业务背景：团期回团、项目完成或服务单完成后，业务侧会沉淀成本事实。财务不能凭口头、微信、Excel 或供应商一句话付款，必须看到成本来源、供应商账单、费用项目、附件和业务对象。

演示流程

  1. 业务侧确认团期实际成本、项目成本、自由行采购批次成本或服务单代垫成本。
  2. 供应商账单通过上传、邮件导入、接口或业务成本承接进入系统。
  3. 财务进入供应商账单，查看供应商、关联团期/项目、账单金额、币种和汇率。
  4. 系统匹配业务成本项；匹配成功进入应付确认，无业务对象进入待挂接，金额不一致进入差异处理。
  5. 进入应付管理，核对应付来源、供应商、费用项目、原币应付、汇率、本币金额、已付、待付、预付冲抵和发票风险。
  6. 确认应付后，才允许进入付款申请。

财务边界：预估成本不是应付；供应商账单确认不等于已经付款；预付也不是成本。财务确认应付后，才进入付款申请和付款执行。

十、场景五：付款申请、审批、付款执行和回单

业务背景：付款链路必须讲清四个动作：业务发起付款申请、审批中心审批、财务执行付款、回单核销。付款申请通过不等于已付款。

金额口径含义
应付金额已确认应付总额。
已申请金额历史已发起付款申请金额。
可申请金额当前还能申请的余额。
本次申请金额本次实际申请付款金额。
已付款金额出纳已经执行成功并回写的金额。
待付金额应付扣除已付款和预付冲抵后的余额。

演示流程

  1. 进入付款申请，查看付款对象、付款依据、关联团期/项目、可申请金额和本次申请金额。
  2. 核对供应商账单、应付单、合同节点、预付冲抵和附件。
  3. 复核通过后进入审批。
  4. 审批通过后进入付款执行。
  5. 出纳核对付款账户、收款账户、账户余额、币种、汇率和付款方式。
  6. 执行付款，上传或同步回单。
  7. 回单核销后更新应付状态，生成 NC 付款任务。

异常处理：超额申请要阻断或特殊审批；账户异常退回补资料；银行失败进入付款退回；回单金额不一致进入回单差异；重复申请要按应付、供应商、付款节点校验。

十一、场景六：团期/项目/门店/渠道结算

结算是财务对业务经营结果的确认，不是业务补录，也不是单纯报表。它统一核收入、成本、退款、应付、付款、发票、回单和毛利。

结算类型核对内容
团期结算团期收入、实际成本、退款、应付、付款、毛利。
项目结算锁定报价、阶段收款、项目成本、变更、毛利。
自由行订单结算订单收入、采购批次成本、供应商对账、订单利润。
单项服务结算服务费收入、代垫款、服务成本、退款扣损。
门店月结门店订单、收款、水单、预存扣款、退款、差异。
渠道/OTA 结算渠道订单、平台账单、佣金扣点、退款、应回款。

业务结算演示

  1. 进入业务结算，按团期、项目、订单或服务单筛选。
  2. 查看结算摘要：收入、成本、已收、待收、已付、待付、毛利。
  3. 核对收入明细：应收、已认款、退款、优惠、待收。
  4. 核对成本明细：供应商、费用项目、应付、预付冲抵、已付、待付。
  5. 核对发票和回单，处理差异。
  6. 确认结算，进入报表和 NC。

门店与渠道对账演示

  • 门店月结：系统汇总门店订单、收款、银联/POS 水单、预存充值、预存扣款、退款，财务处理缺流水、预存未确认、退款未执行、订单状态不一致。
  • 渠道/OTA 对账：导入平台账单后，系统匹配订单、计算佣金扣点、退款折减和应回款，财务处理订单缺失、退款不同步、佣金规则差异、平台扣项异常。

讲法重点：以前门店和渠道月结靠 Excel 汇总；系统目标是自动汇总订单、收款、退款、预存、佣金和扣点，财务只处理差异。

十二、场景七：发票、回单、多币种与汇率

这部分不能从三条主链里拆出去单讲成能力清单，而要作为财务证据链和入账口径贯穿收入、成本付款、结算。

发票与回单

  • 销项开票来自应收、收款、结算和客户抬头。
  • 进项收票来自供应商账单、应付和付款结果。
  • 退款后触发红冲或差额处理。
  • 付款或退款执行后，必须通过回单核销证明资金结果。
  • 发票和回单状态会影响结算风险、NC 和报表口径。

多币种与汇率

概念财务口径
交易币种订单或供应商账单发生时的原始币种。
本位币公司财务核算币种，通常为 CNY。
结算币种实际收款或付款币种。
锁定汇率按业务日、账单日、付款日或结算日确定的核算汇率。
汇率来源第三方 API、银行牌价、人工修正、业务锁定。

培训提醒：多币种不是只显示一个汇率。所有涉及钱的对象都应保留原币金额、汇率、本币金额、汇率来源和锁定时间，否则应收、应付、付款、结算和报表都无法追溯。

十三、场景八：NC 推送与异常处理

NC 是最终入账承接系统。ERP 中已经确认的财务事件，会生成 NC 凭证任务。NC 推送失败不代表业务失败，也不能回滚业务状态。

来源何时生成 NC 任务
收款凭证认款确认后。
退款凭证退款执行完成后。
应付确认应付单确认后按规则生成。
付款凭证付款执行并回单核销后。
发票任务开票、收票、红冲或验真完成后。
结算单结算确认后。
汇兑差异外币结算或关账确认后。

演示流程

  1. 进入 NC 推送，查看待推送、失败和日志。
  2. 打开一张凭证任务，查看来源单据、金额、科目、辅助核算、客户/供应商、币种、汇率。
  3. 对待推送任务执行推送。
  4. 对失败任务查看失败原因。
  5. 根据原因修复科目、辅助核算、主体、币种、发票号、客户供应商档案或凭证字段。
  6. 修复后重新推送，成功后保存 NC 返回号和日志。

十六、月结、报表和管理看板

月结不是月底手工汇总 Excel，而是系统按已发生业务对象自动汇总，财务处理差异。报表不是财务重新统计，而是系统从已生效单据自动沉淀。

报表/看板来源对象作用
应收余额表应收、认款、退款看客户、门店、渠道待收。
收款认款明细到账流水、认款、收款凭证看钱从哪里来、认到哪里。
预收/预存余额表预收、预存账户看余额、冻结、使用。
应付余额表应付、付款、预付冲抵看供应商待付。
付款执行表付款申请、付款单、回单看付款进度和回单状态。
团期/项目毛利表结算单、收入、成本看经营结果。
门店经营报表门店订单、收款、预存、结算看门店经营和月结。
渠道结算报表渠道订单、平台账单、佣金、回款看渠道对账和应回款。
发票统计表发票任务、红冲、税控同步看开票、收票和异常。
多币种汇率差异表外币应收、应付、付款、结算看汇兑损益。
NC 推送报表NC 凭证任务和日志看推送成功、失败、待处理。

管理看板重点：本月应收/已收/待收，本月应付/已付/待付，未认款流水金额，门店和渠道差异金额，供应商预付余额，毛利异常，发票异常，NC 推送失败，关账预检缺口。

十七、讲师演示顺序

  1. 订单确认，系统生成应收。
  2. 银行/银联/门店水单进入到账流水。
  3. 系统自动匹配认款，财务处理未匹配和差异。
  4. 自动生成开票任务，电子发票或税控同步结果回写。
  5. 团期回团，业务提交成本和结算依据。
  6. 供应商账单匹配成本，财务确认应付。
  7. 付款申请审批通过，出纳付款并回单核销。
  8. 业务结算、门店月结、渠道/OTA 对账确认。
  9. NC 推送和失败重推。
  10. 月末关账预检和报表穿透。

十八、必须反复强调的边界

  • 订单提交生成应收，不等于已经收款。
  • 到账流水不是收款凭证，认款确认后才生成收款凭证。
  • 售后审批通过不等于已退款，财务执行退款后才完成。
  • 预估成本不是应付，实际成本或账单确认后才进入应付。
  • 付款申请审批通过不等于已付款，出纳执行成功且回单核销后才算完成。
  • 预付不是成本，必须冲抵实际应付。
  • 发票任务不是结算完成，发票状态只是结算和税务风险之一。
  • NC 推送失败不代表业务失败，只代表凭证还未入 NC。
  • 报表只取已生效单据，不在报表里直接处理业务。
  • 财务不替业务补订单、名单、成本依据和供应商资料，财务负责核对、退回和卡控。
