# 销售部门系统使用培训稿（场景演示版）

## 一、先讲清销售岗位在系统里的位置

销售不是系统里孤立的“下单员”，也不是财务、履约或产品的替代角色。销售在整条旅游业务链路中的位置是：把客户需求识别清楚，把客户愿意购买的产品或方案确认清楚，把订单金额、联系人、游客资料、合同和收款信息提交清楚，让后续履约、财务和售后能够继续承接。

用一句话讲给销售团队：

销售负责把“客户想买什么”变成一张“能执行、能收款、能签合同、能跟踪”的订单。

系统里的销售链路可以按下面这条主线理解：

客户/线索进入 -> 需求判断 -> 匹配标准产品或发起定制询价 -> 报价/下单 -> 订单生成 -> 收款/认款 -> 合同/资料 -> 履约承接 -> 售后变更 -> 门店/财务对账

不同场景的前半段入口不同，但只要形成订单，后半段都会汇合到同一组动作：

- 看订单状态。
- 看应收和收款状态。
- 补游客、证件、名单和出行资料。
- 生成或发送合同/确认书。
- 跟踪履约承接状态。
- 发起改期、转团、退款、转款等售后。
- 处理尾款、认款、异常退回和对账差异。

## 二、培训表达方式

这份材料不按菜单讲，而按场景讲。每个场景统一用七句话说明：

- 客户情境：客户从哪里来，想解决什么问题。
- 销售判断：销售先判断什么，哪些信息必须问清楚。
- 系统入口：从哪个业务对象进入，不按菜单背诵。
- 操作步骤：按真实操作顺序演示。
- 状态变化：意向、订单、收款、合同、资料状态如何变化。
- 下游承接：完成后交给产品、履约、财务还是审批。
- 销售下一步：销售后续回到哪里跟进。

建议培训顺序：

1. 先讲标准产品直接成交，这是所有人的基准流程。
2. 再讲前半段不同入口：新意向、老客户、电销线索、企业定制。
3. 再讲产品类型差异：邮轮、专列、自由行、单项服务。
4. 再讲订单生成后的公共后链路：收款、认款、合同、资料、售后。
5. 最后讲店长/主管每天如何看风险和推动团队。

### 岗位必学清单

为了避免所有人都听成“大而全系统介绍”，培训时要先告诉大家不同岗位听课重点不同。

| 岗位 | 必须掌握 | 重点理解 | 不作为重点 |
| --- | --- | --- | --- |
| 销售顾问/门店店员 | 意向、产品预订、新建订单、订单详情、收款登记、合同发送、游客资料、售后发起 | 订单状态、收款状态、资料缺口、合同状态、售后影响 | 门店成员配置、全店经营报表、财务结算、NC |
| 电销顾问 | 电销意向、新建线索、分配/跟进、转意向订单、转产品预订 | 线索有效性、跟进节奏、高意向判断、转化入口 | 合同细节、门店对账、复杂售后 |
| 门店店长/销售主管 | 门店工作台、顾问订单、顾问意向、尾款、合同、认款、售后异常、门店对账 | 全店风险、顾问转化、预留超时、尾款到期、认款退回、对账差异 | 逐个替店员录游客资料 |
| 大客户/项目销售 | 企业意向、单团询价、项目订单、项目合同、阶段收款、项目名单 | 先询价锁方案，再生成项目订单；阶段款和项目名单是关键 | 把企业团硬塞进普通散客订单 |

### 核心状态流转速查

培训时可以把下面几条状态流转放在屏幕上，让学员先有“系统对象怎么往下走”的整体感。

| 业务对象 | 常见状态流转 | 销售要做什么 |
| --- | --- | --- |
| 意向 | 新意向 -> 跟进中 -> 询价中/高意向 -> 已转订单/已放弃 | 补需求、设跟进、转产品或转询价 |
| 电销线索 | 新咨询 -> 待回访 -> 跟进中 -> 高意向 -> 已转意向订单/已转订单/已关闭 | 确认有效性、跟进、分配、转化 |
| 标准订单 | 预留/占位 -> 待确认 -> 已确认 -> 已完成/已取消 | 收款、补资料、发合同、跟踪履约 |
| 项目订单 | 报价锁定 -> 项目订单 -> 合同待签 -> 阶段收款 -> 项目名单 -> 项目团执行 | 合同、阶段款、名单和客户确认 |
| 收款认款 | 待认领 -> 待复核 -> 已确认/退回补充 | 匹配流水、分配应收、补凭证 |
| 合同 | 待生成 -> 待确认 -> 待审核 -> 待发送 -> 签署中 -> 已生效/已作废 | 核对变量、生成、发送、跟签署 |
| 售后 | 草稿/提交 -> 审批中 -> 已通过/已驳回 -> 财务或履约处理 -> 完成 | 发起申请、补依据、跟踪结果 |

## 三、场景一：客户目标明确，直接成交标准产品

### 客户情境

客户已经明确要报名一个标准产品，比如“欧洲十国经典游，7月中旬出发，2个成人，北京出发，预算两三万以内”。这类场景是销售最常见的直接成交路径。

### 销售先判断什么

销售不要一上来就下单，先确认六件事：

- 客户是谁：新客户、老客户、企业客户还是分销/OTA客户。
- 买什么产品：目的地、产品类型、线路方案。
- 哪天出发：团期、航次、班期或出行日期是否明确。
- 几个人：成人、儿童、老人，是否有特殊价或特殊占位。
- 钱怎么算：价格口径、优惠、定金/尾款、客户需收。
- 能不能执行：余位、资料要求、签证/证件截止、成团或确认规则。

### 系统操作步骤

第一步，从产品预订进入，按客户需求查产品。

销售根据目的地、出发城市、产品类型、出行日期、价格区间、余位状态筛选产品。这里的目标不是浏览所有产品，而是找到“客户现在能买的产品”。

第二步，打开产品详情抽屉，做成交前核对。

重点看：

- 产品线路和行程天数。
- 费用包含和费用不含。
- 最近可售团期/出行日期。
- 余位或库存。
- 起价和价格口径。
- 签证、证件或材料截止。
- 销售状态是否可预订。

讲师可以这样说：这里不是产品介绍页，而是销售成交前的核对页。客户问“还有位吗、多少钱、包含什么、材料什么时候交”，答案都应该从这里确认。

第三步，进入新建订单。

下单方式选择“指定产品”，系统带入产品、线路和可售日期。销售再确认：

- 产品线路或方案。
- 团期/航次/班期/出行日期。
- 成人、儿童、老人或其他人数。
- 房型、舱型、铺位或套餐。
- 联系人和手机号。
- 来源渠道、归属门店、下单销售。

第四步，选择订单处理方式。

根据客户确认程度选择：

- 预留：客户很快付款，系统短时间保留库存。
- 占位：客户基本确定，但还要等主管、资源或客户二次确认。
- 直接确认：客户已经确认并满足扣库存规则。

这里要讲清楚：预留、占位、确认不是随便选的三个按钮，而是销售对“客户承诺程度”和“库存占用责任”的判断。

第五步，选择收款计划。

常见方式：

- 先收定金，尾款到期前认款。
- 一次性收全款。
- 门店预存账户扣款。
- 大客户月结认款。

第六步，生成订单。

生成后系统展示：

- 订单号。
- 产品和团期/服务。
- 联系人。
- 订单应收。
- 优惠减免。
- 客户需收。
- 订单状态。
- 收款状态。
- 下一步入口。

### 状态如何变化

标准成交场景的状态一般这样变化：

产品可预订 -> 生成预留/占位/确认订单 -> 客户付款 -> 销售登记收款或提交认款 -> 财务复核 -> 合同生成/签署 -> 游客资料补齐 -> 履约承接执行

### 下游交给谁

- 产品、价格、团期由产品/运营维护，销售只引用。
- 库存扣减和资源执行由履约、团控或资源运营承接。
- 认款、退款、结算和 NC 由财务承接。
- 销售继续负责客户沟通、资料补齐、合同发送和订单状态跟进。

### 销售下一步看哪里

订单生成后，销售不要停在新建订单结果页，而要回到订单详情：

- 应收收款：看客户待收、已收和认款状态。
- 游客信息：补游客、证件、签证、偏好。
- 合同资料：生成合同、确认书、附件。
- 售后变更：客户改期、退订、退款时发起。
- 操作日志：确认关键动作是否已经记录。

## 四、场景二：客户需求不明确，从意向培育到成交

### 客户情境

客户只表达了模糊需求，比如“暑假想带孩子去欧洲，预算5万左右，不知道跟团还是自由行”。这类客户还不能直接下订单，必须先沉淀成意向。

### 销售先判断什么

销售要把客户的口头需求转成系统能承接的字段：

- 客户是谁，是否已有档案。
- 出行人是谁，大概几人。
- 想去哪里，目的地或主题是什么。
- 时间确定到什么程度：日期未定、只确定月份、日期范围、指定团期。
- 预算范围。
- 偏好和禁忌：亲子、老人、舒适酒店、购物、自费、签证担忧。
- 下次什么时候跟进。

### 系统操作步骤

第一步，新建意向订单。

在意向里先录客户信息：

- 个人客户：客户姓名、手机。
- 企业客户：企业名称、联系人、联系手机。
- 客户来源：门店咨询、官网留资、小程序咨询、电销线索、渠道推荐、大客户转介。
- 归属店员。

第二步，录需求信息。

销售需要补齐：

- 需求主题，例如“暑期欧洲亲子游”。
- 意向品类，例如标准跟团、邮轮、专列、自由行、单项服务、研学、单团项目。
- 目的地/主题。
- 出发城市。
- 出行时间和日期范围。
- 预计人数。
- 预算范围。
- 客户需求描述。

第三步，设置跟进安排。

录下次跟进提醒时间。这个动作的业务意义是：客户暂时没有成交，但不能丢失；销售下一次跟进有依据。

第四步，根据需求成熟度分流。

- 客户已经明确产品：从意向转订单。
- 客户想先比较产品：进入产品预订，匹配标准产品。
- 客户需求超出标准产品：发起单团/定制询价。
- 客户暂不考虑：标记放弃或保留跟进记录。

### 状态如何变化

新意向 -> 跟进中 -> 高意向/询价中 -> 已转订单或已放弃

意向的作用是记录客户需求、归属和跟进节奏。它不是财务对象，也不是履约对象，只有转成订单或询价项目后，后续系统才会继续承接。

### 下游交给谁

- 标准产品成交：进入销售订单。
- 单团、MICE、研学定制：进入定制询价，由计调、产品或项目运营报价。
- 未成交客户：继续由销售跟进。

### 销售下一步看哪里

销售每天应从意向状态看：

- 新意向有没有首次跟进。
- 跟进中客户有没有下次提醒。
- 询价中客户是否已经有报价结果。
- 已转订单客户是否已经完成收款和合同。
- 已放弃客户是否有重新激活价值。

## 五、场景三：老客户复购，不从零开始

### 客户情境

老客户再次咨询，比如曾经报名欧洲线路，现在想看地中海邮轮或北欧长线。老客户不能当成新客户处理，因为系统里已经有历史订单、偏好、证件和风险。

### 销售先判断什么

进入客户档案先看：

- 历史订单：买过什么产品，订单是否完成。
- 客户价值：累计消费、会员等级、积分和券。
- 当前风险：有没有未收款、退款中、售后冻结、积分冻结。
- 证件情况：护照、身份证、有效期、同行游客资料。
- 出行偏好：酒店、座位、饮食、同行人、特殊备注。
- 当前维护顾问和归属门店。

### 系统操作步骤

第一步，进入客户详情。

先看客户摘要：

- 累计消费。
- 历史订单数。
- 未收/退款状态。
- 当前维护顾问。

第二步，看同行游客和证件。

如果客户要买出境游，重点检查：

- 护照是否有效。
- 同行人证件是否齐全。
- 是否存在需要补护照、有效期临近、材料待补。

第三步，看历史订单和收款/合同记录。

如果客户历史上有转团补差、退款中、合同未签或应收差额，要先讲清楚当前是否影响新订单。

第四步，从客户详情进入新建订单或产品预订。

老客户下单时，系统可复用客户和同行人信息，但销售仍然要核对本次出行人、证件、手机号和特殊要求。

第五步，在订单备注和游客资料里带入偏好。

例如：

- 长途飞行优先过道座。
- 饮食少油少辣。
- 希望舒适型酒店。
- 亲子同行需要儿童床或相邻房。

### 状态如何变化

客户档案 -> 产品匹配 -> 新订单 -> 资料复核 -> 收款/合同 -> 履约承接

老客户场景的重点不是多一个页面，而是减少重复录入，同时避免把历史风险带到新订单里。

### 下游交给谁

- 客户归属、会员和资料由客户档案承接。
- 新订单进入销售订单。
- 签证、出行资料、履约特殊要求交给履约或计调。
- 收款和优惠核对交给财务复核。

### 销售下一步看哪里

销售要在客户详情和订单详情之间来回看：

- 客户详情看历史和偏好。
- 订单详情处理本单应收、合同、游客资料和售后。

## 六、场景四：电销线索转销售机会

### 客户情境

客户来自来电咨询、官网留资、小程序、活动名单、老客回访或门店转接。电销的目标不是立刻把所有线索下单，而是把有效线索转成可跟进、可成交的销售机会。

### 电销先判断什么

- 客户是否真实有效。
- 手机号是否能联系。
- 目的地、人数、时间、预算有没有基础信息。
- 意向强度：新咨询、待回访、跟进中、高意向。
- 应该分配给哪个顾问。
- 是转意向订单、转产品预订，还是直接新建订单。

### 系统操作步骤

第一步，新建或接收电销意向。

录入：

- 客户姓名和手机号。
- 目的地。
- 产品类型。
- 意向内容。
- 预算。
- 来源。
- 归属顾问。
- 下次回访时间。
- 登记备注。

第二步，分配顾问。

主管或电销负责人根据目的地、产品类型、顾问能力和当前跟进量分配。

第三步，打开电销意向跟进抽屉。

在抽屉中看：

- 客户需求。
- 跟进记录。
- 本次跟进结果。
- 下次回访。
- 跟进内容。

第四步，选择转化动作。

- 保持跟进：继续预约下一次回访。
- 设为高意向：客户已明确时间、预算或产品。
- 转意向订单：交给销售顾问持续跟进。
- 转产品预订：客户已接近成交，直接去查产品。
- 新建订单：客户已明确购买。
- 关闭意向：无效或暂不出行。

### 状态如何变化

新咨询 -> 待回访 -> 跟进中 -> 高意向 -> 已转意向订单/已转订单/已关闭

### 下游交给谁

电销转出的有效机会进入销售顾问的意向或订单，不再停留在线索池。销售顾问继续完成产品匹配、下单、收款、合同和资料。

### 销售下一步看哪里

销售顾问接到电销转来的客户后，要先看电销跟进记录，避免重复问客户同样的问题，然后再进入意向订单或产品预订。

## 七、场景五：企业客户、MICE、研学或单团项目

### 客户情境

企业客户提出团建、年会、奖励旅游、商务考察、研学营、学校团队等需求。这类场景通常不能直接走标准团期，因为人数、服务内容、预算、资源和合同都需要单独确认。

### 销售先判断什么

- 客户是企业、学校、机构还是个人大团。
- 联系人和决策人是谁。
- 项目类型：MICE、研学、企业包团、定制团、单团项目。
- 人数规模和分组。
- 出行时间和目的地。
- 预算和付款方式。
- 是否需要会议、晚宴、课程、摄影、用车、酒店标准、无购物等特殊服务。
- 是否需要多个报价版本。

### 系统操作步骤

第一步，在意向订单录企业需求。

客户信息要选企业客户，录企业名称、联系人和联系手机。需求主题可以写成“某科技公司年会欧洲行”“学校北京航天研学营”等。

第二步，发起单团/定制询价。

当客户需求不是标准产品能直接满足时，从意向发起询价，把人数、预算、目的地、出行时间、服务要求和客户备注交给定制/计调承接。

第三步，等待报价方案。

报价过程中销售关注：

- 方案版本。
- 服务范围。
- 费用包含和不含。
- 报价金额。
- 可选增项。
- 客户确认状态。

第四步，客户确认方案后生成项目订单。

项目订单不是普通散客团订单，重点是：

- 订单关联单团项目。
- 项目团或客户专属团承接履约。
- 合同金额来自锁定报价。
- 阶段收款计划按首款、进度款、尾款或锁定报价节点承接。

第五步，处理项目合同和阶段收款。

项目合同要核对：

- 企业客户。
- 授权签署人。
- 项目名称。
- 项目团或出行日期。
- 合同金额。
- 锁定报价附件。
- 阶段收款计划。

项目收款认领时，要把到账流水分配到具体阶段应收，比如项目首款、进度款、尾款。

第六步，导入项目名单和交接履约。

项目名单里要维护：

- 姓名、证件、手机号。
- 交通分组。
- 住宿分房。
- 管理层或特殊人员。
- 特殊餐食、会议活动、晚宴或用车要求。

### 状态如何变化

企业意向 -> 单团询价 -> 计调报价 -> 客户确认方案 -> 锁定报价 -> 生成项目订单 -> 项目合同 -> 阶段收款 -> 项目名单 -> 项目团履约

### 下游交给谁

- 报价和资源方案由单团/计调/产品承接。
- 项目订单由销售承接客户和合同。
- 项目团履约由项目运营、计调和资源团队承接。
- 阶段收款和认款由销售提交、财务复核。
- 成本、应付、结算由履约和财务承接。

### 销售下一步看哪里

销售重点看：

- 意向是否已转询价。
- 报价是否已锁定。
- 项目合同是否签署。
- 项目首款是否认款。
- 名单是否导入。
- 项目变更是否影响金额和合同。

## 八、场景六：邮轮订单，重点讲舱房和登船资料

### 客户情境

客户购买邮轮产品，比如理想号地中海邮轮，两间阳台舱，包含成人、儿童或第三/第四人。

### 邮轮销售先判断什么

- 航次和登船日期。
- 舱型、舱房数量、同舱人。
- 成人占位、儿童占位、儿童不占位、第三/第四人。
- 舱号是否已回传。
- 登船资料是否完整。
- Final Payment、DL、船方取消规则。
- 港务费、岸上项目、升舱差价。

### 系统操作步骤

第一步，产品预订中筛选邮轮。

重点看航次、舱型余量和价格矩阵。

第二步，新建订单时选择舱型和人数结构。

邮轮不是简单填“几个人”，而是要按舱房和入住结构判断价格及库存影响。

第三步，订单详情补登船资料。

游客信息里重点维护：

- 护照。
- 英文姓名。
- 国籍、生日。
- 登船表。
- 同舱人。
- 岸上项目选择。

第四步，合同资料处理邮轮合同和舱房补充条款。

第五步，售后时按邮轮库存口径处理。

比如：

- 退整间舱，可能释放舱房。
- 退儿童不占位，只影响金额，不释放舱房。
- 退第三/第四人，调整入住结构，不一定释放整间舱。
- Final Payment 后退订，要等船方确认扣损。

### 状态如何变化

航次可售 -> 选择舱型 -> 生成订单 -> 舱房待确认/已确认 -> 登船资料待补/已齐 -> 船票待出/已出 -> 履约承接

### 下游交给谁

- 舱房确认、船票和船方扣损由邮轮运营或履约承接。
- 销售负责客户确认、资料收集、合同发送、收款和售后发起。
- 财务承接认款、退款和扣损后的退款执行。

## 九、场景七：专列订单，重点讲铺位和实名

### 客户情境

客户购买东方丝路专列，关心软卧、上下铺、同厢人员和上下车站。

### 专列销售先判断什么

- 班期和发车日期。
- 铺位类型和余量。
- 上铺、下铺、同住关系。
- 上下车站。
- 实名资料是否完整。
- 老人、儿童、特殊照顾需求。
- 专列运营方退改规则。

### 系统操作步骤

第一步，产品预订选择专列班期和铺位。

第二步，新建订单时按铺位或包厢确认人数与价格。

第三步，订单详情补实名资料和上下车站。

第四步，履约状态里关注铺位分配、实名核验和出行通知。

第五步，若客户改铺、改班期或退铺，从订单发起售后。

### 状态如何变化

班期可售 -> 铺位占用 -> 实名待核 -> 铺位分配 -> 出行通知 -> 专列履约

### 下游交给谁

铺位分配和运营商确认由专列计调/履约承接。销售负责客户确认、实名资料、收款、合同和售后申请。

## 十、场景八：自由行订单，重点讲服务项确认

### 客户情境

客户购买机票+酒店+接送机+当地玩乐的自由行套餐。自由行订单经常涉及供应商二次确认。

### 自由行销售先判断什么

- 出行日期。
- 酒店、房型、入住人。
- 机票航班、票号、出票状态。
- 接送机航班号和联系人。
- 当地玩乐或单项服务。
- 哪些服务项已经确认，哪些待供应商回传。

### 系统操作步骤

第一步，产品预订筛选自由行产品或套餐。

第二步，新建订单选择套餐、房型、服务项和出行人。

第三步，订单详情看服务确认。

销售要知道：

- 酒店确认号由资源/履约回填。
- 票号由票务或履约回填。
- 接送机信息需要客户提供航班号。
- 当地玩乐确认由供应商或地接承接。

第四步，资料页补入住人、护照/身份证、航班、房型偏好。

第五步，若供应商取消、平台退款、酒店扣损等异常，发起售后。

### 状态如何变化

套餐可售 -> 生成订单 -> 资源待确认 -> 部分确认/已确认 -> 出行资料待补/已齐 -> 履约完成

### 下游交给谁

销售负责客户侧确认；酒店、机票、接送机、当地玩乐由资源/履约承接；成本和应付由履约确认后进入财务。

## 十一、场景九：单项服务订单，重点讲服务单履约

### 客户情境

客户只购买签证代办、材料服务、当地玩乐、接送机、保险等单项服务。

### 销售先判断什么

- 服务项目是什么。
- 办理时效。
- 服务费、材料费、代垫费、加急费。
- 所需材料。
- 客户是否需要授权书或委托协议。
- 服务交付方式。

### 系统操作步骤

第一步，从产品预订选择单项服务，或手工下单录入服务。

第二步，新建服务订单。

订单中重点确认：

- 服务项目。
- 服务单号。
- 服务费或代垫费。
- 联系人。
- 材料清单。

第三步，收款。

单项服务收款要区分：

- 服务费。
- 代垫费。
- 材料费。
- 快递费。
- 补差。

第四步，订单详情维护材料和办理进度。

例如签证代办：

- 护照。
- 照片。
- 申请表。
- 资产证明。
- 授权书。
- 收件、初审、送签、出签、交付。

### 状态如何变化

服务订单 -> 材料待补 -> 已受理 -> 办理中 -> 已交付/已完成

### 下游交给谁

服务办理由服务单承接，不进入团期履约。销售负责客户材料和收款；签证、材料审核、送签、交付由服务运营或履约人员处理；财务处理收款、退款和代垫核对。

## 十二、产品类型差异速查表

这一段适合在讲完几个产品类型场景后做统一回收。学员要记住：系统里“订单”是统一对象，但不同产品的库存、资料和履约承接完全不同。

| 产品类型 | 下单时重点 | 资料重点 | 履约承接 | 售后特别注意 |
| --- | --- | --- | --- | --- |
| 参团游 | 团期、线路方案、成人/儿童/老人、余位、成团规则 | 游客名单、护照/身份证、签证资料、紧急联系人 | 团期执行、签证、出团通知、领队安排 | 转团、退团、签证/机位/酒店扣损 |
| 邮轮 | 航次、舱型、舱房数、同舱人、儿童占位/不占位、第三/第四人 | 护照英文名、登船资料、舱房信息、岸上项目 | 邮轮运营、舱房确认、船票、船方确认 | 退整间舱、退第三/第四人、退儿童不占位的库存影响不同 |
| 专列 | 班期、铺位、上下铺、包厢、上下车站 | 实名资料、身份证、铺位偏好、特殊照顾 | 专列计调、铺位分配、运营商确认、出行通知 | 改铺、改班期、退铺要看铺位和运营商规则 |
| 自由行 | 套餐、酒店、房型、机票、接送机、当地服务 | 入住人、证件、航班、接送机联系人、房型偏好 | 酒店确认、出票、接送机、当地玩乐确认 | 供应商取消、平台退款、酒店/机票扣损 |
| 单项服务 | 服务项目、办理时效、服务费/代垫费/材料费 | 材料清单、授权书、办理资料、交付信息 | 服务单办理、材料初审、送签/送办、交付 | 已送办、已代垫、材料不齐时退款口径不同 |
| 研学 | 营期、课程、学员人数、学校/机构、带队要求 | 学员资料、监护人授权、健康表、保险、安全材料 | 研学执行、课程资源、安全复核、开营通知 | 未成年人安全材料、保险和监护人确认必须留痕 |
| 单团/MICE | 企业客户、项目需求、报价版本、服务范围、阶段款 | 项目名单、交通分组、住宿分房、会议活动需求 | 项目团执行、资源确认、项目成本、客户验收 | 人数、日期、服务范围变化要走项目变更 |

## 十三、所有订单生成后的公共后链路

前面的场景不同，但订单生成后销售都要回到同一条公共链路。

### 1. 订单详情是本单跟进主页面

顶部先看：

- 订单号。
- 产品和出行安排。
- 订单应收。
- 客户已收。
- 客户待收。
- 联系人。
- 归属门店和销售。
- 合同主体。
- 订单状态。
- 收款状态。

### 2. 应收收款

销售在这里看：

- 金额摘要。
- 优惠使用。
- 应收明细。
- 收款/认款记录。

如果客户加购、人数变化、补差、售后减免或人工特批折扣，需要发起应收调整申请。

应收调整不是销售自己改金额，而是提交审批或财务复核。

### 3. 游客信息和资料

销售在这里补齐本单履约需要的客户侧资料：

- 游客名单。
- 证件信息。
- 签证材料。
- 登船资料。
- 实名资料。
- 学员和监护人资料。
- 入住人、航班、接送机信息。
- 特殊偏好和备注。

### 4. 合同资料

销售在这里处理：

- 合同/确认书生成。
- 合同发送。
- 签署状态。
- 行程附件。
- 费用明细。
- 补充协议。
- 出行文件。

合同金额必须来自订单确认金额、锁定报价金额或售后变更差额。

### 5. 售后变更

客户改期、转团、退订、退款、转款、项目变更，都从订单发起售后。销售需要填写原因、影响对象、金额影响和客户确认依据。

### 6. 操作日志

所有关键动作都要能回溯：

- 谁创建订单。
- 谁调整应收。
- 谁登记收款。
- 谁发送合同。
- 谁补资料。
- 谁发起售后。
- 审批和财务反馈是什么。

## 十四、收款与认款专题：用付款场景讲

销售培训中收款不能只讲“点收款按钮”，要按客户怎么付款来讲。

### 场景 A：客户扫码付款

销售在订单收款中选择聚合扫码，系统生成付款码或付款链接。客户付款后，系统可能自动形成支付记录或进入财务复核。

销售要核对：

- 款项是不是本单。
- 本次收款金额是否正确。
- 收后待收是否为零或符合尾款计划。

### 场景 B：客户对公转账

销售先登记或等待银行流水到账，再到收款认领中把流水匹配到订单应收。

操作重点：

- 找到待认流水。
- 核对付款方。
- 核对到账金额和可认金额。
- 选择订单应收。
- 填写本次认款金额。
- 上传银行回单或凭证。
- 提交认款申请。

提交后进入待复核，不等于财务已确认。

### 场景 C：门店现金收款

现金收款要登记：

- 收款门店。
- 收款人。
- 收款时间。
- 付款方。
- 现金收据或交款凭证。

现金仍需要进入认款和财务复核，不能只在线下收了现金不进系统。

### 场景 D：预存账户抵扣

适合门店、企业客户或预存客户。

销售要核对：

- 预存账户。
- 可用余额。
- 本次抵扣。
- 抵扣后余额。
- 是否有冻结金额或财务复核。

### 场景 E：OTA结算认款

平台订单不是简单按客户付款认款，需要上传 OTA 对账单，解析平台订单，核对佣金扣除和退款折减，再提交认款申请。

### 销售和财务边界

- 销售可以登记收款、发起认款、补凭证。
- 财务确认到账、复核认款、处理退款、结算和 NC。
- 销售要跟踪状态：待认领、待复核、已确认、退回补充。

## 十五、售后异常专题：从订单出发，不从菜单出发

售后要用客户真实诉求讲。

### 场景 A：客户要转团或改期

销售从订单详情发起售后，选择转团/改期。

需要确认：

- 原团期或原服务。
- 转入团期或新日期。
- 转出人数。
- 新旧价格差。
- 原名额何时释放。
- 是否生成补差应收。
- 是否需要补充协议。

状态变化：

订单 -> 转团申请 -> 资源确认 -> 补差应收 -> 客户收款/认款 -> 原资源释放/新资源确认

### 场景 B：客户要退团或退订

销售需要确认：

- 退订对象：整单、部分游客、舱房、铺位、服务项。
- 距离出发日期或 DL 的时间。
- 已发生资源成本。
- 供应商扣损。
- 客户确认依据。
- 预计退款。

状态变化：

订单 -> 退订申请 -> 履约确认资源和扣损 -> 审批 -> 退款申请 -> 财务执行

### 场景 C：客户要退款

退款要讲清不是销售直接退款：

- 销售发起退款申请。
- 履约确认扣损。
- 审批确认可退金额。
- 财务执行退款。
- 订单和合同根据结果调整。

### 场景 D：客户要转款

比如客户取消原订单后改报新订单，希望把已付款转到新订单。

销售要确认：

- 原订单已收金额。
- 可转金额。
- 目标订单。
- 转入后目标订单待收变化。
- 客户确认依据。

审批通过后由财务做转款核销。

### 售后列表怎么看

销售在售后列表不只是看“有没有通过”，而要看：

- 售后单号和订单号。
- 客户和人数。
- 产品/团期。
- 金额影响。
- 单据状态。
- 当前节点。
- 后续承接。

如果状态是“已驳回”或“退回补充”，销售要补充客户确认、平台凭证、供应商扣损或其他依据。

## 十六、店长/主管场景：每天看风险，不逐单替销售操作

### 店长每天先看什么

门店店长关注全店经营和风险：

- 本月销售额。
- 本月订单数。
- 顾问成交情况。
- 资金账户余额。
- 今日到账和扣款。
- 尾款到期订单。
- 合同待发送或待签署。
- 客户跟进到期。
- 待认款和退回补充。
- 售后异常。
- 资料缺口。

### 店长如何推动团队

店长不是替每个顾问补资料，而是用系统判断：

- 哪些顾问有高意向未转化。
- 哪些订单预留快到期。
- 哪些尾款影响确认或出行。
- 哪些合同未发或未签。
- 哪些认款退回需要补凭证。
- 哪些售后异常需要销售补充材料。

### 门店对账场景

月度或周期性对账时，店长或门店财务联系人要核对：

- 期初余额。
- 充值入账。
- 订单扣款。
- 退款转预存。
- 管理费。
- 期末余额。
- 差异金额。

核对结果：

- 无异议：确认并提交财务复核。
- 有差异：提交异议。
- 被财务退回：补充充值、扣款、退款或凭证资料后重新提交。

## 十七、现场演示台本

这一节是给讲师使用的。培训现场不要从头念全文，而是按下面七轮演示带学员走。每轮演示都要让学员看见：从哪个业务对象进入、录什么、状态怎么变、后续谁接。

### 第一轮：标准产品成交闭环

演示目标：让所有销售先掌握最标准的一条主流程。

客户设定：赵女士要报名欧洲十国经典游，2个成人，7月中旬出发，预算两三万，准备今天先付定金。

操作路径：

产品预订 -> 筛选欧洲/参团游/北京出发/7月 -> 打开产品详情抽屉 -> 核对线路、费用、余位、材料截止 -> 新建订单 -> 选择产品线路和团期 -> 填人数与价格 -> 填联系人和归属门店 -> 选择预留或确认 -> 选择定金/尾款收款计划 -> 生成订单 -> 进入订单详情 -> 收款 -> 合同资料 -> 游客信息

关键字段：

- 产品线路、团期、出发日期。
- 成人/儿童/老人数量。
- 订单应收、优惠减免、客户需收。
- 联系人、手机号、来源渠道、归属门店、下单销售。
- 处理方式：预留、占位或确认。
- 收款计划：定金、尾款或全款。

状态变化：

产品可预订 -> 订单预留/确认 -> 未收款/部分收款 -> 合同待生成 -> 游客资料待补 -> 履约待承接

讲师提示：

这里要反复强调，销售不是“先把订单建出来再说”。下单前必须确认客户、产品、日期、人数、价格、余位和资料要求。订单生成只是成交开始，不是工作结束。

学员练习任务：

让学员自己完成一张标准跟团订单，要求最后能说清楚这张订单当前还缺什么：是收款、合同、资料，还是履约确认。

### 第二轮：模糊意向到订单

演示目标：让销售理解客户需求不明确时，先沉淀意向，再转产品或订单。

客户设定：李先生只说“暑期想带孩子去日本，2大1小，预算2到5万，不确定跟团还是自由行”。

操作路径：

意向订单 -> 新建意向订单 -> 选择个人客户 -> 录客户姓名和手机 -> 录客户来源和归属店员 -> 录需求主题、意向品类、目的地、出发城市、出行时间、人数、预算、客户需求 -> 设置下次跟进 -> 保存 -> 跟进中 -> 转产品预订 -> 匹配日本亲子/关西/暑期产品 -> 转订单 -> 接入第一轮标准成交闭环

关键字段：

- 需求主题。
- 意向品类。
- 目的地/主题。
- 出行时间。
- 预计人数。
- 预算范围。
- 客户需求。
- 下次跟进提醒。

状态变化：

新意向 -> 跟进中 -> 高意向/询价中 -> 已转订单

讲师提示：

意向不是“还没成交的订单”，而是销售用来管理客户需求和跟进节奏的对象。客户还没确定产品、日期、人数或预算时，不要强行下单。

学员练习任务：

让学员录一个亲子游意向，并判断下一步是继续跟进、转产品预订，还是发起定制询价。

### 第三轮：老客户复购

演示目标：让销售理解老客户不是从零开始，要利用客户档案减少重复确认，同时避开历史风险。

客户设定：张建国是老会员，历史购买欧洲线路，现在咨询地中海邮轮。

操作路径：

散客会员/客户详情 -> 查客户摘要 -> 看历史订单、累计消费、会员等级、未收/退款 -> 看同行游客和证件有效期 -> 看出行偏好 -> 从客户详情进入产品预订或新建订单 -> 选择邮轮航次 -> 带入联系人和归属 -> 在订单资料中补登船资料和同舱人信息

关键字段：

- 当前维护顾问。
- 历史订单。
- 证件有效期。
- 同行游客。
- 出行备注。
- 未收/退款状态。

状态变化：

客户档案可用 -> 新订单生成 -> 资料复核 -> 收款/合同 -> 履约承接

讲师提示：

老客户场景的价值是“先看历史，再做新单”。如果客户有未收款、退款中、证件待补，要先处理或提示风险。

学员练习任务：

让学员进入一个客户档案，找出是否可以直接下新订单，并说明本次成交要复用哪些信息、还要补哪些资料。

### 第四轮：企业定制项目

演示目标：让销售理解企业/MICE/研学不能直接套普通散客订单，要先询价和锁定方案。

客户设定：北京某科技有限公司计划80人欧洲年会奖励旅游，需要会议、晚宴、摄影、商务舱和五星酒店。

操作路径：

意向订单 -> 新建企业意向 -> 录企业客户、联系人、人数、预算、出行时间、特殊需求 -> 发起单团/定制询价 -> 进入单团项目详情查看报价方案 -> 客户确认报价版本 -> 生成项目订单 -> 处理项目合同 -> 登记阶段收款/收款认领 -> 导入项目名单 -> 项目团履约承接

关键字段：

- 企业客户和授权联系人。
- 项目名称。
- 人数和分组。
- 预算。
- 服务范围。
- 报价版本。
- 锁定报价金额。
- 阶段收款计划。
- 项目名单。

状态变化：

企业意向 -> 单团询价 -> 报价中 -> 报价锁定 -> 项目订单 -> 项目合同待签 -> 项目首款待认款 -> 项目名单待导入 -> 项目团执行

讲师提示：

企业定制项目的核心是“先方案，后订单”。销售不能为了快，把企业需求直接建成普通手工订单，否则后续合同、阶段收款、名单、资源和成本都承接不清。

学员练习任务：

让学员判断三个需求：2人欧洲团、80人企业年会、30人研学营，分别应该直接下单、建意向，还是发起定制询价。

### 第五轮：收款认款

演示目标：让销售分清收款登记和财务确认。

客户设定：客户已对公转账25,600元，银行流水已到账，但订单仍显示待认款。

操作路径：

订单详情 -> 查看应收收款 -> 收款认领 -> 搜索流水号/付款方/订单号 -> 选择待认流水 -> 分配订单应收 -> 填本次认款金额 -> 上传银行回单 -> 提交认款申请 -> 查看状态进入待复核 -> 回订单详情查看收款记录

关键字段：

- 流水号。
- 到账金额。
- 可认金额。
- 付款方。
- 订单号。
- 应收金额。
- 已认金额。
- 本次认款金额。
- 银行回单。

状态变化：

待认领 -> 待复核 -> 已确认/退回补充

讲师提示：

销售提交认款后，状态是待复核，不代表钱已经财务确认。被退回时，销售要补流水、付款方、回单或订单匹配依据。

学员练习任务：

给学员一个流水和两个可能订单，让他们判断应该匹配哪个订单、认多少金额、是否有差额。

### 第六轮：售后变更

演示目标：让销售理解售后会影响资源、金额、合同和财务，必须从订单发起并留痕。

客户设定：客户报名欧洲团后要求改到下月同线路团期，新团期每人贵600元，共4人，需要补差2,400元。

操作路径：

订单详情 -> 售后变更 -> 发起售后 -> 选择转团/改期 -> 选择原团期和目标团期 -> 填转团人数、原因、价差、客户确认依据 -> 提交审批 -> 查看售后列表 -> 审批通过后生成补差应收 -> 收款/认款 -> 原团名额释放/新团名额确认

关键字段：

- 原订单号。
- 原团期。
- 目标团期。
- 转团人数。
- 补差金额。
- 客户确认依据。
- 资源影响。
- 合同或补充协议。

状态变化：

售后申请 -> 审批中 -> 资源确认 -> 补差应收 -> 待收款 -> 完成

讲师提示：

销售不能口头承诺“可以退多少、能不能改成”。系统里要先走售后，履约确认资源和扣损，财务再承接补差或退款。

学员练习任务：

让学员分别判断：改期、整单退订、部分游客退款、已收款转新订单，各应该选择哪类售后。

### 第七轮：店长/主管风险视角

演示目标：让主管知道每天如何用系统看全店风险，而不是逐单替销售操作。

客户设定：门店今天有尾款到期、合同待签、认款退回和售后异常。

操作路径：

门店工作台 -> 今日需关注 -> 查看尾款、合同、待认款、售后异常 -> 进入顾问订单或顾问意向 -> 按顾问查看订单和跟进状态 -> 门店详情查看销售与合同规则 -> 门店对账查看余额、扣款、充值和差异 -> 提交确认或异议

关键字段：

- 本月销售额。
- 本月订单数。
- 资金账户余额。
- 今日到账和扣款。
- 尾款到期。
- 合同待发送/待签署。
- 认款退回。
- 售后异常。
- 对账差异。

状态变化：

风险提醒 -> 进入对应订单/意向/认款/售后 -> 顾问补处理 -> 店长复核进度 -> 财务或履约承接

讲师提示：

店长看的是全店经营和风险，店员看本人客户和订单。培训店长时不要重复讲每个按钮，而要讲如何发现风险、定位责任人、推动处理。

学员练习任务：

给店长一个“今日需关注”列表，让他判断哪三件事优先处理，以及分别应该让哪个角色处理。

## 十八、培训中的常见错误提醒

这些错误建议在每一轮演示结束后穿插提醒，比最后集中念更有效。

| 常见错误 | 为什么不对 | 正确做法 |
| --- | --- | --- |
| 客户还没确定日期、人数和预算就直接下单 | 订单金额、库存和合同都不稳定 | 先建意向，补需求后再转产品或订单 |
| 只看产品起价，不核对团期、余位和价格口径 | 容易报价不准或客户误解 | 在产品详情和新建订单里核对最终客户需收 |
| 企业团建直接建普通手工订单 | 项目报价、阶段收款、名单和合同承接不清 | 先走企业意向和单团询价，报价锁定后生成项目订单 |
| 客户对公转账后只口头说已付款 | 订单仍然无法确认收款状态 | 到收款认领匹配流水并提交认款申请 |
| 合同金额和订单应收不一致 | 后续认款、发票、退款和结算都会出问题 | 合同金额必须来自订单或锁定报价 |
| 游客资料只存在聊天记录里 | 履约无法办签证、出票、分舱、分铺 | 在订单详情维护游客、证件、材料和附件 |
| 口头承诺退款金额 | 扣损、规则和财务执行未确认 | 从订单发起售后，等审批和扣损确认 |
| 认款退回后不处理 | 订单可能一直显示未收或部分收款 | 补付款方、流水、回单或匹配依据后重新提交 |
| 店长只看销售额，不看待收和售后 | 高销售额可能伴随资金和履约风险 | 同时看尾款、合同、认款、资料和售后异常 |

## 十九、销售人员必须记住的边界

- 产品能不能卖，由产品和运营维护；销售负责引用和判断客户适配。
- 库存能不能确认，由订单处理方式和履约确认共同决定；销售负责发起预留、占位或确认。
- 客户有没有付款，销售可以登记或提交认款；财务最终确认。
- 订单金额变化，销售发起应收调整；审批和财务复核后生效。
- 合同金额必须来自订单或锁定报价；销售不能线下改金额绕过系统。
- 游客资料由销售收集，履约用来办签证、出票、分舱、分铺、发通知。
- 退款和扣损由售后流程承接；销售不能直接承诺最终可退金额。
- 门店余额和对账由门店确认，财务复核后形成最终口径。

## 二十、培训收口

销售使用系统，不是为了“把页面点完”，而是为了让每一个客户需求在系统里有归属、有状态、有金额、有资料、有合同、有收款、有后续承接。

培训最后可以用这句话收口：

销售每天在系统里只盯四类事：

- 哪些客户还没转化。
- 哪些订单还没收款或合同没签。
- 哪些资料缺口会影响出行。
- 哪些售后和认款需要补材料。

只要销售把这四类事处理清楚，产品、履约、财务就能沿着同一张订单继续往下走。
