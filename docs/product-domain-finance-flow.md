# 产品域与业财流程总模型基线

更新时间：2026-06-29

本文是凯撒旅游 SaaS 产品域与业财流程改造的总基线。后续字段标准、页面改造、导航归属、财务单据和 NC 推送模板，都必须能回到本文定义的业务对象和数据链路。

## 1. 总目标

让凯撒旅游 SaaS 的产品对象、库存对象、订单对象、成本对象、收付对象、结算对象、NC 凭证对象形成一条可追溯的数据链路。

每个页面、字段、按钮和流程必须回答六个问题：

1. 当前页面维护哪个业务对象。
2. 这个对象从哪里来。
3. 这个对象流向哪里。
4. 是否影响库存、订单、成本、收款、付款、退款、结算。
5. 是否触发财务单据。
6. 是否进入 NC 推送或税务台账。

## 2. 产品域总分层

系统产品域分为五条链路：

```text
标准线路：资源 -> 线路 -> 方案 -> 团期 -> 订单 -> 收款/执行/成本 -> 结算单 -> 应付/付款 -> NC
MICE单团：需求 -> 报价版本 -> 锁定报价 -> 唯一团期/项目 -> 订单/收款 -> 项目结算 -> NC
自由行：套餐组合 -> 出行日期价格库存 -> 订单 -> 采购批次 -> 订单结算 -> NC
单项服务：服务项目 -> 服务单 -> 执行跟进 -> 服务费/代垫拆分 -> 按单结算 -> NC
外采产品：外采来源 -> 包装/定价/配额 -> 按原产品类型销售 -> 差额核算 -> NC
```

这五条链路最终都进入统一财务中心，但进入财务前的业务锚点不同：

| 链路 | 核心业务锚点 | 库存锚点 | 订单锚点 | 结算锚点 | 收入口径 |
|---|---|---|---|---|---|
| 跟团 | 团期 | 人数名额 | 销售订单 | 团期结算单 | 销售全额 |
| 邮轮 | 团期 | 舱型/舱位 | 销售订单 | 团期结算单 | 销售全额 |
| 专列 | 团期 | 车厢/铺位/座位 | 销售订单 | 团期结算单 | 销售全额 |
| 研学 | 团期 | 学生/班级/导师配置 | 销售订单或学校订单 | 团期结算单 | 销售全额 |
| MICE单团 | 项目/唯一团期 | 项目资源清单 | 项目订单/合同 | 项目结算单 | 锁定报价金额 |
| 自由行 | 订单 + 采购批次 | 出行日期价格库存 | 销售订单 | 订单结算单 | 销售全额或差额，取决于供给模式 |
| 单项服务 | 服务单 | 服务能力/办理时效 | 服务单 | 服务单结算 | 服务费为收入，代垫为代收代付 |
| 外采 | 原产品类型 | 跟随原产品类型 | 销售订单 | 跟随原产品类型 | 差额收入 |

## 3. 标准线路模型

跟团、邮轮、专列、研学统一采用“线路 -> 方案 -> 团期”三层模型。

### 3.1 线路层

线路层是长期稳定的售卖对象，回答“这个产品是什么”。线路层不维护具体出发日期、当期价格和当期库存。

线路层维护：

- 产品名称、线路类型、旅游类型、目的地。
- 适合人群或适合年级。
- 产品亮点标签。
- 封面图、详情图、附件文档。
- 发布渠道、上架时间。
- 通用费用包含、费用不含、自费推荐、儿童/老人或学生/老师费用说明、退改政策。

### 3.2 方案层

方案层是线路下的差异化售卖结构，回答“这条线路如何组织服务”。一条线路可以有多个方案。

方案层维护：

- 跟团：方案名称、行程天数、出发城市、成团规模参考、分日行程、酒店、航线/车型、POI、餐厅、交通配置。
- 邮轮：船公司、船名、船舶信息、舱型结构、靠港行程、岸上游、登离船交通。
- 专列：承运方、列车号、经停站、车厢/铺位/座位结构、下车游览、地面接驳。
- 研学：教育目标、课程体系、合作基地、合作学校/机构、带队老师要求、导师配置、每日课程、安全预案。

方案层只维护结构，不维护当期销售库存。

### 3.3 团期层

团期层是某个方案在某个出发日期的执行实例，回答“哪一天卖、卖多少钱、还剩多少、谁来执行”。团期不重复录入完整行程，只引用方案并补充当期变化。

团期层维护：

- 关联线路、关联方案、团号。
- 出发日期、回程日期、出发城市/港口/车站。
- 当期最大容量、最小成团人数、预留名额。
- 当期价格、早鸟价、尾期特卖价、渠道差价。
- 可售渠道、开售时间、截止售卖时间。
- 领队、集合地点、集合时间、特别注意事项。

不同产品的库存矩阵不同：

- 跟团：按人数维护成人、儿童占床、儿童不占床、老人、单房差。
- 邮轮：按舱型维护库存和价格，舱型来自方案层。
- 专列：按铺位/座位维护库存和价格，铺位/座位来自方案层。
- 研学：按学生容量、班级数量、带队老师、研学导师配置维护。

## 4. 非标准线路模型

### 4.1 自由行

自由行可以在旅游线路列表中作为类型展示，但底层不进入团期模型。

自由行链路：

```text
套餐配置 -> 出行日期价格库存 -> 订单下单 -> 采购执行 -> 采购批次对账 -> 成本分摊 -> 订单结算
```

自由行核心对象：

- 套餐组合：机票 + 酒店、机票 + 酒店 + 接送机、纯酒店、当地玩乐组合。
- 出行日期价格库存：某套餐在某日期的价格、酒店房量、可售人数、截止时间。
- 采购批次：按出行日期、目的地、套餐类型、供应商归集订单，用于供应商对账。
- 订单结算单：以订单为结算单位，批次成本分摊后核算单订单盈亏。

自由行关键规则：

- 不出现在团期列表，不使用“开排团期”命名。
- 采购批次是成本对账锚点，订单是利润核算锚点。
- 没有采购批次时，供应商对账和单订单盈亏会断链。

### 4.2 单项服务

单项服务不是线路，不进入“新建线路”。它承接签证代办、机票代订、酒店代订、接送机、保险代购等可售服务。

单项服务链路：

```text
服务项目配置 -> 服务单创建 -> 执行跟进 -> 服务费确认 -> 代垫成本核销 -> 按单结算
```

单项服务核心对象：

- 服务项目：服务名称、材料要求、办理时效、收费方式、默认流程。
- 服务单：客户、服务内容、办理节点、执行责任人、完成时间。
- 服务费：凯撒真实收入。
- 代垫款：代收代付资金，不计入营业收入。
- 服务单结算：按服务单核算服务费收入、直接成本、代垫核销和退款扣损。

单项服务关键规则：

- 主命名使用“单项服务/服务单”，不使用“单项委托”作为主菜单或主对象名称。
- NC 推送时服务费和代垫款必须走不同会计科目。

### 4.3 MICE 单团自组

MICE 和定制团是项目型成团流程，不是标准线路库。

MICE 链路：

```text
需求录入 -> 报价制作 -> 报价确认锁定 -> 生成唯一团期或项目执行单 -> 收款/执行/成本 -> 项目结算
```

MICE 核心对象：

- 需求单：客户、人数、日期、预算、特殊要求。
- 报价版本：方案 A/B/C，可导出 PDF，可记录发送和反馈。
- 锁定报价：客户确认后金额锁定，成为应收基准。
- 项目/唯一团期：进入执行、成本、结算。
- 项目结算单：以锁定报价为应收基准，核对实收和实际成本。

MICE 关键规则：

- 财务应收基准来自锁定报价，不来自标准团期价格。
- 后续如需要变更报价，必须通过变更记录或变更单，不直接覆盖锁定报价。

### 4.4 外采产品

外采不是独立产品结构，而是供给来源。外采产品的结构跟随原产品类型。

外采链路：

```text
线路市场引入 -> 外采原始产品 -> 包装方案 -> 采购价维护 -> 销售价/渠道价 -> 配额申请 -> 销售下单 -> 差额结算
```

外采核心对象：

- 外采原始产品：供应商原始报价、行程、采购价、配额规则。
- 包装产品：凯撒侧重命名、描述、封面、亮点、渠道价。
- 配额：供应商给定团期、舱型、铺位、日期或服务能力的可售额度。
- 差额结算：销售价减采购价形成凯撒差额收入。

外采关键规则：

- 外采跟团仍按团期结算，外采邮轮仍按舱型结算，外采自由行仍按订单和采购批次结算。
- 客户收款可以按销售全额入账，但收入确认不能按全额确认。
- NC 需要支持全额收款、采购价付款、差额收入结转、采购成本结转。

## 5. 数据流主轴

标准数据流：

```text
资源主数据 -> 线路 -> 方案 -> 团期 -> 销售订单 -> 应收/收款 -> 执行 -> 成本确认 -> 结算单 -> 应付 -> 付款执行 -> NC推送
```

成本支线：

```text
团期/项目/订单 -> 预估成本 -> 实际成本 -> 成本确认 -> 应付草稿 -> 应付确认 -> 付款申请 -> 付款执行 -> 回单 -> NC推送
```

退款支线：

```text
订单 -> 售后申请 -> 审批 -> 退款执行 -> 应收冲减/预收冲减 -> 退款凭证 -> NC推送
```

发票支线：

```text
订单/收款/结算 -> 销项开票 -> 税务台账 -> NC或税控接口
应付/付款/供应商账单 -> 进项收票 -> 认证状态 -> 税务台账
```

发票是并行业务链路，不固定排在付款执行之后。

## 6. 业财事件基线

每一笔财务单据都必须由业务事件触发，不允许脱离业务对象事后补录。

| 业务事件 | 触发财务动作 | 财务对象 | NC 影响 |
|---|---|---|---|
| 订单提交 | 生成应收或预收 | 应收单/预收记录 | 待收款，暂不推送或按预收规则推送 |
| 认款完成 | 确认到账 | 收款凭证 | 生成 NC 收款凭证 |
| 资金池扣款 | 扣减门店资金池余额 | 收款凭证/资金池流水 | 生成 NC 收款或内部结算凭证 |
| 改价/补差 | 调整应收 | 应收调整单 | 收款后推送 NC |
| 退差 | 冲减应收或生成退款 | 应收冲减/退款单 | 推送红冲或退款凭证 |
| 退款执行完成 | 实际退款 | 退款凭证 | 生成 NC 退款凭证 |
| 回团成本确认 | 确认实际成本 | 成本凭证/应付草稿 | 生成 NC 成本凭证或待结算推送 |
| 应付确认 | 确认供应商应付 | 应付单 | 进入付款申请，不一定立即推送 |
| 付款申请通过 | 准备执行付款 | 付款申请 | 暂不推送 |
| 付款执行完成 | 实际付款 | 付款单/回单 | 生成 NC 付款凭证 |
| 结算完成 | 核算收入、成本、毛利 | 结算单 | 生成 NC 结转凭证 |
| 销项开票 | 开票给客户 | 发票台账 | 更新税务台账，按接口规则推送 |
| 进项收票 | 收供应商发票 | 发票台账 | 更新税务台账，按接口规则推送 |

## 7. 模块职责边界

资源模块：

- 维护 POI、酒店、餐厅、车型、航线、锁位、邮轮资源、专列资源、供应商、领队资源。
- 只维护可复用基础资源，不维护某个具体产品的销售结构。

产品中心：

- 维护线路、方案、自由行套餐、单项服务项目、外采包装、团期、定价策略、渠道授权。
- 团期团控承载团期列表、成本付款、团队执行、回团处理。

订单中心：

- 承载线路浏览与报价、报价单、意向订单、订单管理、收款认领、合同管理、售后处理。
- 转团、改签改期、退款申请属于订单中心售后。

财务：

- 承载收款、认款、应收、应付、付款、退款执行、对账结算、发票、回单、资金池、汇率币种、NC 推送、财务报表。
- 退款执行属于财务，退款申请属于订单售后。

审批：

- 承接线路发布、价格调整、渠道授权、成本、付款、退款、折扣、转团、结算等跨模块审批。

渠道中心：

- 承载 OTA、分销商、代理商、佣金规则、同步日志。
- 分销商、代理商不是供应商，不挂到资源模块。

## 8. 状态机基线

标准团期不使用一个字段承载所有状态，应拆成三类状态：

| 状态组 | 状态 | 用途 |
|---|---|---|
| 销售状态 | 筹备中/开放报名/已截止/售罄/已取消 | 决定是否可售和是否占用库存 |
| 执行状态 | 未出团/操作中/已出发/已回团/已归档 | 决定执行资料、名单、通知、回团动作 |
| 结算状态 | 未维护成本/预估成本/实际成本确认/结算中/已结算 | 决定成本确认、结算单和财务报表 |

关键触发：

- 开放报名：销售侧可见并可下单。
- 截止报名：锁定名单，不再接受新订单。
- 已回团：开启实际成本录入和回团处理。
- 实际成本确认：自动生成结算单草稿。
- 已结算：推送结转凭证，盈亏进入报表。
- 已取消：触发全团或相关订单退款流程，并处理预付款冲抵或退回。

## 10. 第一阶段验收标准

本文作为第一步基线，验收标准如下：

1. 五条产品链路已经明确。
2. 标准线路和非标准线路的核心业务锚点已经明确。
3. 线路、方案、团期三层边界已经明确。
4. 自由行不进入团期模型，采购批次作为成本对账锚点。
5. 单项服务不进入线路模型，服务费和代垫款分账。
6. 外采作为供给来源，不作为万能产品类型。
7. 业财事件已经能覆盖收款、退款、成本、应付、付款、结算、发票和 NC。
8. 后续页面改造有统一任务卡模板。

下一步进入“对象字典”建设，逐一定义资源、线路、方案、团期、订单、服务单、采购批次、结算单、应收、应付、付款申请、NC 凭证等对象的主键、上游、下游和责任模块。
