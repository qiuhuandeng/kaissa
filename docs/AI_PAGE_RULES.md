# AI 页面制作执行规则

本文件用于约束 AI 后续修改凯撒旅游 SaaS 原型页面时的执行方式。它不是完整业务说明，而是每次动手前必须读取并落实的短规则。

## 1. 修改前先判断业务位置

任何页面、抽屉、弹窗、Tab、按钮、列表、详情页修改前，先回答六个问题：

1. 这个页面给谁用。
2. 这个角色当天最高频任务是什么。
3. 上游对象从哪里来。
4. 当前页面处理的核心业务对象是什么。
5. 操作完成后改变哪个状态或生成哪个下游入口。
6. 下游交给谁继续处理。

判断顺序固定为：

角色任务 -> 业务对象 -> 操作频率 -> 状态流转 -> 下游影响 -> 页面入口

不能先看旧页面有什么就补什么。

## 2. 原型优先级

当前项目是业务原型，不是生产系统。

- 优先表达角色操作流程、对象归属、状态含义和下一步入口。
- 不默认做自动回写、跨页面深度联动、真实单据生成、NC 推送闭环。
- 当前页面只做当前角色需要判断和处理的内容。
- 财务确认、付款、结算、发票、NC 等深度动作回到财务模块表达。
- 旧结构不符合业务链路时，按正确业务流程重构，不做补丁式堆字段。

## 3. 页面入口原则

- 低频、稳定、可复用资料放资源、配置或主数据页面。
- 高频运营动作放最常使用它的角色工作路径上。
- 财务确认动作放财务模块。
- 审批动作放审批中心，并回到来源单据展示结果。
- 团期名额、航班锁位、邮轮舱房、专列铺位等动态供给，优先放团期、执行或资源调度视角，不把资源模块当日常处理主入口。

## 4. 列表页结构

后台业务列表优先使用以下结构：

- 外层使用 `.list-surface`。
- 搜索区使用 `.list-surface-filter` 或已有 `.filter-card` 规范。
- 表格容器使用 `.list-surface-table` 或 `.table-wrap`。
- 业务表格优先使用 `.business-optimized-table`。
- 需要明确列宽时使用 `colgroup` 或语义列 class。
- 操作列必须固定右侧，并设置明确宽度。

列表页只展示用于识别对象、判断状态、发现异常和完成高频处理的字段。完整信息进入详情、抽屉、日志或二级页面。

## 5. 列表字段规则

- 一级列表只有一个主状态。
- 如果 Tab 已承载主状态，表格内原则上不重复展示同一主状态。
- 当前待办、风险提醒、临期提醒只做提示、筛选、排序，不驱动操作按钮。
- 按钮由业务对象主流程状态决定。
- 不展示无法被系统字段、数量、单据号、状态或缺口校验的人工作业提醒。
- “关键缺口/当前提醒”只用于复杂流程列表，不是普通列表标配。
- 系统内部编码默认不进一级列表。
- 履约核对编号和钱相关编号必须保留。
- 数量型跳转只让数字本身是蓝色链接，单位和说明保持普通文本。

## 6. 表格排版规则

- 表格容器可以横向滚动，但不能撑破页面外壳。
- 操作列固定右侧，不能被剩余宽度撑宽。
- 多余宽度给业务信息列，不能给操作列。
- 单元格最多三层内容：主信息、辅助信息、一个标签或补充。
- 数量、金额、百分比、状态、编号、操作按钮等关键短值必须完整显示。
- 线路名、客户名、供应商名、备注等长文本才允许省略。
- 一级业务列表单元格默认垂直居中。
- 表格多行内容不要用 `<br>` 叠加块级元素。

推荐单元格结构：

```html
<td>
  <span class="cell-main">丝路专列深度游</span>
  <span class="cell-sub">北京-乌鲁木齐 / 2026-08-15 出发</span>
</td>
```

## 7. 列表操作列规则

列表操作分三类：

- 当前对象可直接完成的快速处理：打开独立抽屉或确认弹窗。
- 当前角色高频下游入口：进入团期、执行、成本、收款、售后等承接页面。
- 低频动作：放入“更多”。

禁止把列表行操作设计成“跳转详情页某个 Tab 继续处理”的半截入口。复杂处理统一放详情页或明确的处理页面。

## 8. 抽屉与弹窗结构

右抽屉和业务弹窗遵守固定层级：

- 标题区只识别动作。
- 内容区识别业务对象。
- 字段区承载业务判断。
- 底部只完成操作。

标题区只放当前动作标题和关闭按钮，例如“新增会员”“团期订单概览”“应付差异处理”。不放模块路径、副标题、说明文案、状态标签、业务摘要。

业务对象信息放内容区：

- 处理类、核对类、详情类抽屉优先使用 `.drawer-object-summary`。
- 主数据档案类详情可使用轻量对象识别条。
- 分区标题使用 `.drawer-section-head` + `.drawer-section-title`。
- 历史表单标题 `.form-section-title` 只用于兼容，新增结构优先用 drawer section。

## 9. 抽屉文案规则

抽屉和业务弹窗默认不写解释性文案。

删除或避免以下类型文案：

- “用于……”
- “这里只……”
- “当前抽屉不……”
- “保存后可……”
- “本抽屉只确认……”

分区标题只写业务名词，例如“基础信息”“证件与偏好”“重点订单”“差异依据”“本次处理”。

业务上下文不是说明文案。供应商、费用项目、关联线路、团期、出发日期、客户、渠道、金额、数量、差异、待收、缺口等，应该拆成摘要字段或识别信息。

## 10. 抽屉类型选择

动手前先判断抽屉类型：

- 创建/编辑/配置类：直接进入表单，已有复杂对象才放轻量摘要。
- 详情/速览/记录类：先识别对象，再放明细表、记录、日志或时间线。
- 业务处理/状态推进类：对象摘要 + 判断依据 + 本次处理项。
- 核对/对账/差异处理类：摘要突出核对结论和核心差异。
- 选择/筛选/关联/分配类：来源对象识别 + 搜索筛选 + 可选列表/本次选择项。

底部主按钮必须使用明确业务动作词，例如“确认资源”“提交申请”“发送通知”“确认调整”。

## 11. CSS 修改规则

- 不随手在 `shared/style.css` 末尾追加散乱样式。
- 新增页面级样式必须有页面前缀。
- 公共列表样式优先复用 `.list-surface` 和 `.business-optimized-table`。
- 公共抽屉样式优先复用 `.drawer-modal`、`.drawer-object-summary`、`.drawer-section`。
- 新增通用能力先判断是否已经有现成 class。
- 避免新增依赖列序的 `td:nth-child(n)` 样式。
- 避免新增 `!important`，除非是兼容历史冲突且局部可控。
- 不为单页临时问题污染全局基础组件。

## 12. 新增或修改页面的交付检查

完成后至少检查：

- 当前角色、对象、上游、下游是否清楚。
- 列表主状态是否唯一。
- 列表按钮是否由主流程状态驱动。
- 操作列是否固定右侧且宽度合理。
- 关键短值是否完整显示。
- 多行单元格是否没有乱用 `<br>`。
- 数量跳转是否只蓝数字。
- 抽屉标题区是否只放动作标题。
- 抽屉内容区是否有必要的对象识别。
- 是否删除解释性文案。
- 是否新增了无前缀、难复用、污染全局的 CSS。
