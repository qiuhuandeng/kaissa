# 系统全链路验收主控方案

## 1. 验收目标

本次验收判断凯撒旅游 SaaS 原型从业务使用和业财规范上是否跑得通，不以真实生产数据联动为目标。

验收主线：

资源主数据 → 产品/方案/外采/单项服务 → 团期/航次/班期/营期/项目团/服务单 → 订单/合同/售后 → 应收/收款/认款 → 成本/应付/付款 → 回团结算/对账结算 → NC 推送。

验收要回答：

| 问题 | 判断标准 |
|---|---|
| 链路是否通 | 用户能从上游对象进入下游处理页，且参数和对象语义正确。 |
| 入口是否顺手 | 高频操作在最常用角色页面，低频配置和反查不抢主入口。 |
| 口径是否正确 | 同一业务动作在列表和详情打开同源抽屉或同类处理内容。 |
| 业财是否可承接 | 业务页表达财务承接方向，财务页可反查业务来源，但不互相越权。 |
| 是否符合原型边界 | 不要求真实跨页动态回写，不为了模拟数据闭环扩大开发量。 |

## 2. 防跑偏规则

后续每一步验收只记录或修正“链路断点、对象错误、入口错误、抽屉口径错误、业财越界”。

不做以下事情：

- 不因为某个页面样式小问题进入单页 UI 打磨。
- 不为了验收做大量动态数据联动。
- 不把业务页面改成财务处理页。
- 不把财务页面改成名单、资源、行程维护页。
- 不把旧页面已有字段当成正确业务结构。

发现问题时按优先级记录：

| 优先级 | 定义 | 是否立即修 |
|---|---|---|
| P0 | 链路断、按钮空、跳错页面、对象类型错误。 | 当前验收阶段可直接修复。 |
| P1 | 抽屉内容业务口径错误、列表/详情同名动作不同源。 | 记录后按模块确认整改。 |
| P2 | 入口效率差、按钮重复、命名不统一、状态驱动不合理。 | 纳入体验整改清单。 |
| P3 | 视觉细节、文案、排版密度。 | 最后统一处理。 |

## 3. 验收产品类型

| 产品类型 | 产品源头 | 履约对象 | 订单对象 | 结算对象 |
|---|---|---|---|---|
| 出境游/国内游 | 旅游线路 → 方案 | 团期 | 游客订单/团队订单 | 回团结算/团期结算 |
| 邮轮 | 邮轮资源 → 邮轮线路/方案 | 航次/舱房 | 邮轮订单 | 航次/船票结算 |
| 专列 | 专列资源 → 专列线路/方案 | 班期/铺位 | 专列订单 | 班期/铺位结算 |
| 自由行 | 套餐/组合产品 | 出行日期/服务确认 | 自由行订单 | 订单/供应商结算 |
| 单项服务 | 服务产品 | 服务单 | 服务订单 | 服务结算 |
| 研学 | 研学线路/营期 | 研学营期 | 研学订单 | 营期/项目结算 |
| 外采产品 | 外采来源/包装 | 外采团期/配额/服务批次 | 外采订单 | 外采结算 |
| 单团自组 | 定制需求/报价 | 项目团 | 项目订单 | 项目结算 |

## 4. 每条链路的验收节点

每个产品类型按同一套节点验收：

| 节点 | 验收重点 | 典型断点 |
|---|---|---|
| 资源源头 | 资源归属、字段边界、是否只做基础资料。 | 资源页承接订单、付款、NC 等动态动作。 |
| 产品/方案 | 线路层、方案层、团期层字段是否分层。 | 在线路层维护具体出发日、价格、库存。 |
| 履约对象 | 团期/航次/班期/项目团/服务单语义是否正确。 | 项目团混入公开渠道余量、OTA 可售等散拼语义。 |
| 订单/合同 | 订单类型、合同类型、名单资料、售后入口是否一致。 | 项目订单打开普通散拼收款或游客报名抽屉。 |
| 应收/收款/认款 | 应收计划、收款登记、认款复核能反查来源。 | 收款页不知道订单类型，项目阶段款混成普通团费。 |
| 成本/应付/付款 | 成本确认、供应商拆分、付款申请、付款执行边界清楚。 | 业务侧直接付款或财务侧维护资源。 |
| 结算 | 业务侧补依据，财务侧确认结算口径。 | 未验收、未确认成本时仍显示可结算。 |
| NC | 只在财务 NC 页面推送、重试、处理失败。 | 业务页面直接推 NC。 |

## 5. 执行步骤

| 步骤 | 名称 | 输出 |
|---|---|---|
| 1 | 固化验收目标和边界 | 本主控方案。 |
| 2 | 菜单与页面对象地图 | 菜单、页面、业务对象、角色、上下游表。 |
| 3 | 产品类型链路矩阵 | 8 类产品从源头到 NC 的路径表。 |
| 4 | 全系统硬检查 | 脚本、链接、按钮、旧词、导航归属检查结果。 |
| 5 | 按产品类型跑链路 | 每条链路的通过项、断点、风险。 |
| 6 | 业财专项验收 | 应收、收款、认款、应付、付款、结算、NC 的承接口径。 |
| 7 | 问题分级和整改建议 | P0/P1/P2/P3 清单。 |
| 8 | 最终报告 | 完整验收报告和下一阶段整改路线。 |

## 6. 最终报告结构

最终报告按以下结构输出：

1. 总体结论。
2. 全链路覆盖范围。
3. 各产品类型链路验收结果。
4. 页面跳转和按钮断点清单。
5. 业财承接口径问题清单。
6. 高优先级整改建议。
7. 可暂缓问题和原因。
8. 后续实施顺序。
