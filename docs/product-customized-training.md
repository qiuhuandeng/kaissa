# 产品部门-定制产品系统使用培训稿（场景演示版）

## 一、先讲清定制产品岗位在系统里的位置

定制产品不是标准产品池，也不是销售订单的简单变体。它围绕一个具体客户、一个具体项目、一个具体出行需求展开。系统里的核心对象不是“线路”，而是“单团项目”。

用一句话讲给定制产品团队：

定制产品负责把“客户的一次性需求”变成“可报价、可锁定、可成团、可执行、可结算”的项目。

系统里的定制产品链路可以按下面这条主线理解：

客户需求/销售意向/引用产品线路 -> 发起单团询价 -> 计调接单或退回补充 -> 新建报价版本 -> 成本测算和对客报价 -> 提交销售确认 -> 销售跟进客户 -> 锁定报价 -> 确认成团 -> 项目订单/项目合同/项目团/阶段收款 -> 项目执行 -> 回团验收 -> 成本结算承接

这里要先讲清：定制产品不是先建一个长期可售产品再排团期，而是先有客户需求，再围绕这个客户项目不断报价、确认、锁定和承接。

## 二、培训表达方式

这份材料按场景讲，不按菜单讲。每个场景统一说明：

- 客户情境：客户要什么，需求从哪里来。
- 角色判断：销售、计调、产品负责人分别判断什么。
- 系统入口：从哪个业务对象进入。
- 操作步骤：按真实操作顺序演示。
- 状态变化：项目、报价、成团和执行状态怎么变化。
- 下游承接：订单、合同、项目团、收款和结算怎么接。
- 下一步：当前角色后续回到哪里处理。

建议培训顺序：

1. 先讲单团项目主线。
2. 再讲询价需求从哪里来。
3. 再讲计调接单和退回销售。
4. 再讲报价版本、成本测算和对客报价。
5. 再讲销售确认、客户反馈、锁定报价。
6. 再讲确认成团后生成的承接对象。
7. 最后讲变更、执行、回团和结算。

### 岗位必学清单

| 岗位 | 必须掌握 | 重点理解 | 不作为重点 |
| --- | --- | --- | --- |
| 定制产品/单团计调 | 待计调接单、退回销售、新建报价版本、成本测算、提交销售确认 | 报价版本和客户确认版本的区别，锁定前后变更规则不同 | 客户收款认款、财务 NC 推送 |
| 项目销售 | 发起单团询价、补充需求、确认客户版报价、记录客户反馈、锁定报价 | 先锁报价再成团，不能口头确认直接执行 | 维护供应商成本明细 |
| 产品负责人/业务负责人 | 查看项目状态、审核报价口径、判断毛利和资源风险 | 报价是否可交给客户，锁定后是否允许成团 | 替计调逐项询价 |
| 履约/项目执行配合人 | 查看项目团、名单、资源确认、出团通知 | 成团后项目团才进入执行链路 | 在报价阶段提前做结算 |

### 核心状态流转速查

| 业务对象 | 常见状态流转 | 当前角色要做什么 |
| --- | --- | --- |
| 单团询价 | 草稿 -> 已提交 -> 待计调 | 销售补齐客户、日期、人数、预算、需求和计调组 |
| 单团项目 | 待计调 -> 报价中 -> 待销售确认 -> 销售跟进中 -> 待确认成团 -> 已成团 -> 执行中 -> 已回团 -> 已结算 | 按状态判断下一步由谁处理 |
| 报价版本 | 草稿 -> 待销售确认 -> 销售跟进中 -> 已锁定/已替换/已作废 | 计调维护，销售确认，客户确认后锁定 |
| 项目承接 | 未生成 -> 项目订单/项目合同/项目团/阶段收款已生成 | 成团后销售、履约、财务分别承接 |
| 变更记录 | 锁定后新增 -> 待确认/已确认 -> 关联报价/订单/合同/项目团 | 锁定后金额、人数、日期和服务变更必须留痕 |

## 三、场景一：销售拿到客户定制需求，发起单团询价

### 客户情境

某科技公司准备组织 80 人去欧洲办年会和奖励旅游，客户只说了大概目的地、日期、预算和活动要求。这个需求不能直接下标准产品订单，需要先形成单团询价。

### 销售先判断什么

销售需要先确认：

- 客户是谁：企业客户还是个人客户。
- 业务类型是什么：MICE、定制团、企业团建、奖励旅游、私人包团、政企考察、研学定制。
- 出行日期是否明确。
- 预计人数和人数结构。
- 目的地、天数和出发地。
- 预算口径：总预算、人均预算或暂未明确。
- 大交通、酒店、用车、餐饮、活动、导游/领队要求。
- 是否引用现有产品或线路。
- 期望报价时间和计调组。

### 系统入口

入口有三类：

- 从单团项目列表点击“新建单团询价”。
- 从销售意向转入单团询价。
- 从自营产品详情点击“发起单团询价”，引用现有产品线路。

培训时要讲清：前半段入口不同，但提交后都会汇合到“单团项目”。

### 操作步骤

第一步，填写客户与项目。

包括客户类型、客户名称、联系人、联系电话、业务类型、需求主题、询价项目名称。

第二步，填写出行需求。

包括出发地、目的地、日期明确度、出行日期、出行月份、天数、预计人数、人数结构、房间需求、预算金额/范围、大交通口径、会议活动口径和预算弹性。

第三步，填写服务与报价要求。

如果引用现有产品，填写引用产品、引用线路、沿用内容和调整内容。如果没有引用，就按客户需求填写酒店标准、用车标准、餐饮标准、导游/领队、会议/活动、门票/体验、特殊要求和补充需求。

第四步，填写内部交办。

包括销售负责人、协同销售、计调组、指定计调负责人、期望报价时间、紧急度和内部备注。

第五步，保存草稿或提交询价。

资料不齐时保存草稿；提交后项目进入待计调。

### 状态如何变化

询价草稿 -> 提交询价 -> 单团项目待计调

### 下游交给谁

提交后交给定制产品/计调接单。销售继续负责客户沟通和补充需求，计调负责资源询价和报价版本。

## 四、场景二：计调接单，判断是否可以进入报价

### 业务情境

单团项目进入“待计调”。计调打开项目详情，发现客户日期、人数和预算都有，但酒店标准和会议活动要求还不够清楚。

### 计调先判断什么

计调要先判断：

- 需求能不能报价。
- 日期、人数、目的地和预算是否足够明确。
- 是否需要引用标准产品线路。
- 服务标准是否会影响成本：酒店、用车、餐饮、活动、导游/领队。
- 期望报价时间是否可实现。
- 是否需要退回销售补充。

### 系统入口

从单团项目列表的“待计调”进入项目详情，或在列表快速处理“计调接单”。

### 操作步骤

第一步，打开项目详情看需求快照。

先看项目摘要：

- 项目编号。
- 询价单号。
- 客户。
- 出行日期和人数。
- 销售/计调。
- 当前报价。
- 项目状态。

第二步，进入询价需求页签。

重点核对：

- 客户与来源。
- 出行需求。
- 服务要求与附件。
- 需求补充记录。

第三步，决定接单或退回销售。

如果需求足够，点击计调接单，填写计调负责人和期望报价时间，项目进入报价中。

如果需求不够，点击退回销售，选择退回原因，如日期待确认、预算待确认、服务标准待补充、客户主体待确认，并填写需补充内容。

### 状态如何变化

待计调 -> 计调接单 -> 报价中  
待计调 -> 退回销售补充 -> 待计调/待补充

### 下游承接

接单后由计调进入报价版本维护；退回后由销售补充需求再提交。

### 讲师提醒

计调不要为了推进状态而勉强接单。定制项目的报价质量取决于需求质量，需求不清会导致多轮返工、毛利失真和客户口径反复。

## 五、场景三：计调新建报价版本，完成资源与成本测算

### 业务情境

计调已经接单，需要做 V1 报价。项目可能引用了标准产品线路，也可能完全手工方案。

### 计调先判断什么

- 这次报价是首次报价、客户反馈调整，还是资源变动调整。
- 报价依据是引用产品线路、供应商方案，还是手工方案。
- 服务方案是否能被客户理解。
- 供应商成本是否有有效期。
- 对客报价是否覆盖成本并满足毛利要求。
- 付款节点是否和客户付款预期一致。
- 哪些供应商报价附件需要留存。

### 系统入口

在项目详情的“报价”页签点击“新建报价版本”。

### 操作步骤

第一步，填写报价基础。

包括版本号、方案名称、报价类型、报价有效期、报价依据、币种/税费口径、报价人和报价日期。

第二步，维护服务方案。

包括引用线路/方案、服务标准、客户差异要求，以及行程安排。行程不是为了好看，而是让销售和客户知道本版报价包含什么服务。

第三步，维护资源与成本测算。

在成本项里填写：

- 成本项。
- 供应商/来源。
- 数量和单位。
- 成本单价。
- 预算成本。
- 有效期。
- 备注。

计调要说明：成本有效期非常重要，客户确认晚了可能要重新报价。

第四步，维护对客报价。

对客报价按客户能理解的口径填写，例如国际交通、酒店及地接服务、项目服务费。系统会汇总客户报价合计、预算成本合计和预计毛利。

第五步，维护条款与附件。

包括费用包含、费用不含、取消及变更规则、报价附件、供应商报价附件和付款节点。

第六步，保存草稿或提交销售确认。

草稿状态可以继续编辑。提交后项目进入待销售确认。

### 状态如何变化

报价中 -> 新建 V1 草稿 -> 保存草稿  
报价中 -> 提交销售确认 -> 待销售确认

### 下游交给谁

提交销售确认后，销售核对客户版报价口径。计调暂不直接锁定报价，除非销售确认客户接受。

## 六、场景四：销售确认报价，客户反馈后多版本调整

### 业务情境

计调提交 V1 报价，销售发现客户需要隐藏内部成本、付款节点需要调整，或者客户反馈希望从四星升级为五星。

### 销售先判断什么

- 报价内容是否能对客。
- 费用包含/不含是否清楚。
- 付款节点是否可接受。
- 报价有效期是否足够。
- 毛利和成本风险是否需要负责人复核。
- 客户反馈是价格问题、服务标准问题，还是日期/人数变化。

### 系统操作步骤

第一步，在待销售确认状态打开报价。

查看报价版本、方案名称、报价总额、预算成本、预计毛利、报价有效期和版本状态。

第二步，确认客户版报价或退回计调调整。

如果可以对客，点击确认客户版报价，项目进入销售跟进中。  
如果不能对客，点击退回计调调整，说明报价口径、服务内容、付款节点或毛利复核要求，项目回到报价中。

第三步，生成客户报价单。

客户报价单读取当前报价版本、服务方案、报价明细、费用规则和付款节点。销售可用于发送客户。

第四步，记录客户沟通。

客户反馈后，记录沟通结果、沟通时间、沟通内容和沟通材料。

第五步，计调复制版本做 V2/V3。

客户反馈导致调整时，不直接覆盖旧版本。复制旧版本形成新版本，再写本次调整原因，旧版本可以被替换或作废。

### 状态如何变化

待销售确认 -> 销售确认 -> 销售跟进中  
待销售确认 -> 销售退回 -> 报价中  
销售跟进中 -> 客户反馈调整 -> 新建/复制报价版本 -> 待销售确认

### 下游承接

销售只把客户确认的版本推进到锁定。没有客户确认依据的报价，不应进入成团。

## 七、场景五：客户确认后锁定报价

### 业务情境

客户通过邮件确认 V3 豪华五星版，价格、人数、日期、服务标准和付款节点都已接受。此时销售要锁定报价。

### 销售先判断什么

锁定前必须确认：

- 当前版本是否是客户确认版本。
- 报价有效期是否仍有效。
- 金额和付款节点是否清楚。
- 客户确认联系人是谁。
- 确认方式是什么：客户邮件、盖章确认单、企业微信、电话确认、合同确认。
- 是否有确认材料。

### 系统操作步骤

第一步，在销售跟进中选择报价版本。

只有销售已确认的对客报价版本可以锁定。草稿和待销售确认版本不能锁定。

第二步，打开锁定报价。

系统展示：

- 锁定版本。
- 锁定金额。
- 报价有效期。
- 付款节点。

第三步，填写客户确认。

包括确认联系人、确认方式、确认时间、确认材料和确认备注。

第四步，确认锁定。

锁定后系统形成锁定快照。快照包括版本、金额、付款节点和客户确认依据。

### 状态如何变化

销售跟进中 -> 报价已锁定 -> 待确认成团

### 下游承接

锁定报价是成团前最关键的业务依据。后续项目订单、合同、项目团和阶段收款都以锁定报价为准。

### 讲师提醒

锁定报价不是“客户大概同意”。没有确认联系人、确认方式和确认时间，就不能把项目推进成团。

## 八、场景六：确认成团，生成项目承接对象

### 业务情境

报价已经锁定，客户确认执行。定制产品负责人或有权限人员确认成团。

### 角色先判断什么

确认成团前要核对：

- 是否已有锁定报价。
- 项目金额是否正确。
- 付款节点是否正确。
- 出行日期和人数是否仍一致。
- 是否需要退回报价调整。
- 成团后哪些对象要被创建或承接。

### 系统操作步骤

第一步，点击确认成团。

系统展示项目、客户、锁定报价、出行、项目金额和付款节点。

第二步，看成团后承接对象。

确认成团后会承接：

- 项目订单：承接客户、金额、名单、售后和订单应收。
- 项目合同：承接签署和客户盖章确认。
- 项目团：承接名单、资源、领队、出团通知和执行。
- 阶段收款计划：承接项目首款、进度款、尾款。

第三步，确认成团或退回报价调整。

如果金额、日期、人数或服务标准有问题，退回报价调整。无问题再确认成团。

### 状态如何变化

待确认成团 -> 确认成团 -> 已成团  
待确认成团 -> 退回报价调整 -> 报价中

### 下游承接

成团后，不再只在单团项目里处理。系统会出现四个下游对象：

- 销售继续跟项目订单和项目合同。
- 履约进入项目团处理名单、资源和出团准备。
- 财务根据阶段收款计划做收款和认款。
- 回团后业务侧形成结算依据，再交给财务结算。

## 九、场景七：成团后进入项目团执行

### 业务情境

项目已成团，出行日期临近。项目团需要导入名单、确认资源、生成出团通知，并把执行状态交给履约。

### 角色先判断什么

- 项目订单是否已生成。
- 项目合同是否进入签署。
- 阶段首款是否按约收款。
- 项目团是否有名单。
- 交通、酒店、会务、用车、领队等资源是否确认。
- 出团通知是否具备生成条件。

### 系统操作步骤

第一步，在项目详情进入项目承接。

查看项目订单、项目合同、项目团和阶段收款计划。

第二步，进入项目团。

项目团承接执行，包括名单、证件、资源、领队、出团通知和成本。

第三步，处理执行准备。

可以继续从单团项目列表的执行处理入口进入：

- 导入名单。
- 资源确认。
- 生成出团通知。
- 进入团期成本。

第四步，回团后上传验收材料和复核成本。

项目已回团时重点看客户验收材料、实际成本、结算依据和结算承接。

### 状态如何变化

已成团 -> 执行中 -> 已回团 -> 已结算

### 下游承接

项目团执行由履约承接，财务只承接确认后的应收、应付、付款、结算和 NC。定制产品/计调要保证报价和资源依据能被执行与结算使用。

## 十、场景八：锁定报价后发生变更

### 业务情境

客户锁定报价后，临时把人数从 80 人改为 92 人，或要求增加一场晚宴。这时不能随意改报价版本，要新增变更记录。

### 角色先判断什么

- 变更发生在锁定前还是锁定后。
- 变更类型是什么：人数、日期、服务标准、金额、付款节点、客户主体。
- 变更是否影响金额。
- 是否需要重新报价或审批。
- 变更会影响报价、订单、合同、项目团还是收款计划。

### 系统操作步骤

第一步，在项目详情进入变更记录。

锁定报价后才显示变更记录。未锁定前通过报价版本调整即可。

第二步，新增变更。

填写：

- 变更类型。
- 影响金额。
- 确认状态。
- 关联单据。
- 变更前。
- 变更后。
- 变更原因。

第三步，根据影响范围承接。

- 人数变更影响项目团名单、资源和报价。
- 日期变更影响交通、酒店和活动资源有效期。
- 服务标准变更影响成本和合同。
- 金额变更影响订单和阶段收款。
- 客户主体变更影响合同和发票。

### 状态如何变化

已锁定 -> 新增变更 -> 待确认/已确认 -> 关联报价/订单/合同/项目团

### 讲师提醒

锁定后的变更不是备注。它是后续争议、收款、成本和结算的依据。

## 十一、定制项目列表怎么看

定制项目列表不是普通订单列表，它帮助团队判断项目卡在哪个阶段。

| 列表信息 | 看什么 | 业务判断 |
| --- | --- | --- |
| 项目/客户 | 项目名称、客户、询价号 | 先识别当前项目 |
| 出行需求 | 日期、目的地、人数、预算 | 判断是否临近和复杂度 |
| 销售/计调 | 谁负责客户，谁负责报价 | 判断当前该找谁 |
| 询价进展 | 需求提交、计调接单、报价提交、客户跟进 | 判断流程卡点 |
| 最新报价 | 版本、金额、成本、毛利、付款 | 判断是否可对客或可锁定 |
| 主状态 | 待计调、报价中、待销售确认、销售跟进中、待确认成团等 | 决定操作 |
| 操作 | 接单、报价、确认、锁定、成团、执行、结算 | 进入当前主流程 |

## 十二、现场演示台本

### 第一轮：建立单团项目主线

打开单团项目列表，从待计调、报价中、待销售确认、销售跟进中、待确认成团、已成团一路讲状态。

讲师话术：

定制产品的核心不是维护一个长期产品，而是推动一个客户项目从询价走到成团和执行。

### 第二轮：新建单团询价

点击新建单团询价，演示：

1. 客户与项目。
2. 出行需求。
3. 服务与报价要求。
4. 内部交办。
5. 提交询价。

### 第三轮：计调接单或退回销售

打开待计调项目详情，演示：

1. 看需求快照。
2. 点击计调接单。
3. 再演示另一种情况：退回销售补充需求。

### 第四轮：新建报价版本

进入报价页签，演示：

1. 报价基础。
2. 服务方案。
3. 资源与成本测算。
4. 对客报价。
5. 条款与付款节点。
6. 保存草稿和提交销售确认的区别。

### 第五轮：销售确认和客户反馈

演示待销售确认状态：

1. 确认客户版报价。
2. 生成客户报价单。
3. 记录客户沟通。
4. 客户提出调整后复制版本。

### 第六轮：锁定报价

在销售跟进中选择报价版本，演示锁定报价：

1. 核对锁定版本和金额。
2. 核对付款节点。
3. 填客户确认联系人、方式、时间、材料。
4. 确认锁定。

### 第七轮：确认成团

演示确认成团弹窗：

1. 看项目、客户、锁定报价、出行和项目金额。
2. 看成团后承接对象。
3. 确认成团。
4. 进入项目承接查看项目订单、项目合同、项目团和阶段收款。

### 第八轮：执行和变更

从已成团或执行中项目演示：

1. 进入项目团。
2. 导入名单。
3. 资源确认。
4. 生成出团通知。
5. 锁定后新增变更。
6. 回团后进入结算承接。

## 十三、常见错误提醒

| 错误做法 | 为什么错 | 正确做法 |
| --- | --- | --- |
| 客户需求不清也接单报价 | 报价反复、成本失真 | 先退回销售补充关键需求 |
| 用备注记录客户反馈 | 后续无法追溯版本 | 记录客户沟通或复制报价版本 |
| 直接覆盖旧报价 | 无法说明为什么金额变化 | 保留版本，复制后形成 V2/V3 |
| 没有客户确认就锁定报价 | 成团依据不足 | 填客户确认联系人、方式、时间和材料 |
| 锁定后继续改报价明细 | 破坏锁定快照 | 通过变更记录处理 |
| 成团前不核对付款节点 | 阶段收款计划错误 | 锁定报价前确认付款节点 |
| 把项目团执行都留在单团项目里处理 | 履约承接不清 | 成团后进入项目团处理名单、资源和通知 |
| 回团后直接让财务结算 | 业务依据不足 | 先补验收材料和实际成本，再交财务 |

## 十四、培训收口

定制产品培训结束时，学员要能回答五个问题：

- 当前项目处于哪个主状态？
- 现在应该由销售、计调、产品负责人、履约还是财务处理？
- 当前报价版本是草稿、待销售确认、销售跟进中，还是已锁定？
- 成团后生成了哪些承接对象？
- 锁定后发生变化，应该走报价版本还是变更记录？

最后一句话：

定制产品不是把客户需求写成一张报价单，而是让一次性项目从需求、报价、客户确认、成团、执行到结算都能有依据地流转。
