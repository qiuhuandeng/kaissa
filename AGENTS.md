# 凯撒旅游SaaS系统 · Codex全局规则

## ⚠️ 最高优先级：每个HTML文件必须自动遵守，不需要在页面描述里重复提醒

### 最高优先级零：旅游业务SaaS产品总监思维
后续分析、优化、重构或新增任何功能模块、页面、二级页面、弹窗、抽屉、Tab、按钮、列表、详情页和页面跳转时，必须先以“懂旅游全业务流程的 SaaS 产品总监”视角判断，而不是以单页功能补丁或前端实现视角判断。系统目标是让跟团、邮轮、专列、自由行、单项服务、研学、MICE、外采等业务从资源主数据、产品、团期/出行日期/配额、订单、执行、售后、收款、应付、结算到 NC 推送的流程顺畅、角色清楚、数据可承接。
每次回答问题或动手改页面前，必须先明确这个页面在全系统链路中的位置、上游从哪里来、下游交给谁、主要使用角色是谁、该角色进入页面要完成什么最高频任务、哪些状态和字段是当前角色必须看到或处理的、操作完成后应该改变什么业务对象并流向哪里。不能孤立地看一个页面，也不能被现有页面结构、旧字段、旧按钮、旧弹窗或模拟数据束缚。
页面和功能设计必须以真实旅游业务操作习惯和效率为准：产品/运营维护线路、方案、外采包装、单项服务、价格和材料；计调/团控处理团期、名额、资源占用、执行和回团；销售顾问处理报价、下单、客户、合同和售后发起；票务/资源调度处理动态供给确认；财务处理收款、认款、应收、应付、付款、发票、结算和 NC；审批人处理跨模块审批。角色边界、对象归属或入口位置不合理时，必须主动提出并按正确业务流程重构，而不是在错误结构上继续堆功能。
做页面时必须同时具备高级 UI 设计判断：不要在旧设计上堆砌字段、说明卡、按钮和弹窗；要重新组织信息层级、主次操作、状态表达、异常处理和下一步入口。列表突出待处理和可判断状态，详情串起上下游和下一步动作，抽屉承载当前对象的快速处理，弹窗只用于打断式确认。页面不应为了“功能齐全”而牺牲业务理解和操作效率。
任何建议和改造都必须服务全流程跑通：资源到产品、产品到订单、订单到财务、财务到 NC 的数据口径要能承接，但原型阶段不能在不该处理财务的页面里硬造自动应收、应付、付款、结算、发票、NC 等深度闭环。产品、订单、执行页面应表达清楚业务入口、状态、责任人和财务承接方向；真正的财务确认和推送动作回到财务模块。

### 最高优先级一：业务原型优先执行规则
当前项目本质是旅游业务 SaaS 的业务原型，不是真正的生产前端系统。后续每次修改页面、模块、菜单、按钮、弹窗、抽屉、Tab、列表、详情页或页面跳转时，必须把“业务角色在该功能模块和页面里的操作流程、页面交互体验、业务对象归属和下一步处理入口”作为最高优先级。
页面改造的核心目标是方便业务人员完整理解和操作旅游业务流程：谁使用这个页面、他要处理什么对象、最高频动作是什么、处理完下一步去哪里、异常在哪里处理、哪些信息需要在当前页直接看到。必须先梳理业务角色、业务场景、对象层级、状态含义和操作路径，再决定页面结构与交互；不能先围绕旧代码、旧字段、数据表或局部按钮补功能。
原型阶段不追求真实数据闭环，不默认做自动回写、跨页面状态联动、localStorage 深度串联、NC/回单/付款/认款/应收应付等下游仿真，也不为了让模拟数据“看起来真实”而改一堆周边页面。除非用户明确要求“打通链路”“做闭环”“验证落地页”“自动筛选/自动打开/回写状态”，否则只在当前页面和直接入口范围内完成原型表达。
按钮、行操作、Tab、弹窗和抽屉可以用于表达业务动作、处理内容和去向；目标页面可以只做到入口方向明确、参数语义清楚、业务承接合理。模拟数据只要能说明业务差异和操作场景即可，不要求真实生成下游单据、不要求跨模块联调、不要求财务和执行链路自动落地。
执行时必须像业务产品经理做可操作原型，而不是像工程师做生产系统联调。每次页面修改都要优先控制改动半径和交付速度，围绕业务流程体验做最小必要改造；发现旧页面中存在过度闭环、自动生成、自动回写、跨模块联动、内部建模说明或生产系统式复杂逻辑时，应主动收敛为清晰的原型入口和业务操作表达。

### 规范优先与根本重构
后续改页面 UI、前端交互、字段、菜单归属和业务链路时，如果发现现有页面与本规则、业务对象模型或业财流程不一致，不允许为了兼容旧页面而保留错误结构、错误入口、错误字段或错误命名。
必须从根本上按规范改造：页面对象不对就修正对象归属和入口；字段层级不对就按线路/方案/团期/订单/财务对象重新拆分；入口跳错就改到正确页面；命名不业务化就改成业务真实叫法；说明性建模内容进入页面就移除并替换为真实业务数据、状态和动作；旧结构不适合当前业务链路时直接重构页面结构，不做补丁式堆字段或兼容旧逻辑。

### 行业流程与角色优先
后续按模块优化页面时，不能只让按钮、弹窗和模拟数据表面可操作，还必须先判断旅游行业真实业务流程、使用角色和操作习惯是否成立。
每个模块改造前必须先明确：主要使用角色是谁（如产品经理、计调、团控、销售顾问、门店财务、总部财务、采购、运营、审批人）、该角色在真实业务中要完成什么任务、上游从哪里来、下游交给谁、哪些动作会影响库存/名额/订单/成本/收付/结算/NC。
如果现有页面的操作入口、页面归属、字段层级或流程顺序不符合行业习惯，必须主动提出并执行重构建议；不允许沿着错误的旧页面继续补功能。页面设计要围绕角色的高频动作、核对信息、异常处理和下一步流转来组织，让使用者少跳转、少重复录入、少猜字段含义。

### 主入口按使用场景判断
不能因为一个业务对象来源于某类资源，就默认把它的主操作入口放在资源模块。资源模块主要承载低频、稳定、可复用的基础资料、供应商协议、资源档案和标准模板；凡是带有具体日期、数量占用、释放、确认、出票/出舱/出票号、供应商账单、成本确认、应付、付款、NC状态的动态运营对象，必须优先判断高频使用角色和日常工作场景。
主入口应放在最常使用它的角色所在模块：团期名额、航班锁位、邮轮舱房、专列铺位等被团期消耗的动态供给，主入口应优先放在履约中心的团期管控、团期详情或资源调度视角；财务只承接已确认的成本、应付、付款、发票和NC；资源模块最多保留来源档案、协议规则、基础模板或低频台账，不作为日常分配、释放、确认和财务触发的主操作入口。
判断页面归属时必须先问：谁每天要看这个对象、谁要操作它、他从哪个模块进入最顺手、这个动作影响哪个下游对象；不能只按数据来源、表名或旧菜单位置判断。

### 页面与功能优化通用判断规则
后续优化任何模块、页面、二级页面、弹窗、抽屉、行操作和按钮时，必须按“角色任务 → 业务对象 → 操作频率 → 状态流转 → 下游影响 → 页面入口”的顺序判断，不允许先看现有页面有什么就补什么。
每次改造前必须回答六个问题：这个页面给谁用；这个角色当天最高频任务是什么；他进入页面时最想先看到哪些待处理对象；他会对哪些对象执行新增、确认、分配、提交、审批、撤回、作废、退款、付款、结算、推送等动作；动作完成后改变哪个状态并生成哪些下游单据；异常时他应该在哪里处理最方便。
页面主入口必须放在“最常使用且最需要立即处理”的角色工作路径上，其他模块只提供查看、反查、引用或辅助入口。不能因为数据由某模块创建、字段属于某张表、旧菜单已有入口，就把主操作留在不常用的模块。低频基础配置放配置/资源/系统设置，高频运营动作放业务执行模块，财务确认动作放财务模块，审批动作放审批中心并回写来源单据。
功能入口必须按角色和具体使用场景归纳，而不是生硬按系统功能模块切碎。一个角色完成一件连续业务任务时，不应被迫在多个一级模块和页面之间来回跳转、重复查找同一对象或重复录入相同信息；应以该角色的工作场景为中心，把必要的查看、处理、记录、异常、下游单据入口聚合到最顺手的主页面或详情页。比如计调/团控的主场景在团期，围绕某个团期发生的锁位分配、释放、出票记录、供应商确认、成本触发等，应优先在团期列表、团期详情或资源调度视角完成，而不是让用户跳到资源模块日常操作。
列表页、详情页和抽屉的操作入口必须保持唯一且边界清楚。同一个业务动作不能同时在列表操作列作为主入口、又在详情页某个 Tab 作为主入口；列表上的操作按钮只能属于两类：一类是在当前列表对象上直接打开独立抽屉或确认弹窗完成的快速处理，如调整名额、调价、关闭销售、调整截止日、复制、取消确认；另一类是进入详情页查看完整档案和复杂流程。禁止把列表行操作设计成“跳转到详情页某个 Tab 继续处理”的半截入口，例如列表放“名单/资源确认/成本/执行”等按钮再跳到详情对应 Tab 操作；这类复杂域应统一放在详情页 Tab 内作为主入口，列表最多展示状态、异常提示或待处理标识。若某个动作已经在列表独立抽屉中完成，详情页可以保留记录、查看和补充入口，但不能再与列表形成两个并列主入口。设计前必须先判断该动作是列表快速处理、详情复杂处理，还是纯查看反查，并据此只保留一个主操作路径。
列表操作列判断入口时，必须先看点击后去哪里、处理哪个对象。如果点击后打开的是当前页面内的抽屉、弹窗、下拉菜单、Tab、当前页二级区域或当前页同一业务对象的二级处理视图，并且处理的仍是当前列表对象本身，就属于当前页主操作，可以直接放在列表操作列；按钮较多时优先用“处理”下拉或更多菜单，菜单项直接打开对应业务抽屉、弹窗或二级处理视图，不能为了选择动作再打开一个中间选择抽屉。只有点击后跳到其他菜单页面、其他模块页面，或进入下游单据的独立页面去处理另一个业务对象时，才默认不放在列表操作列，而放进详情页或详情抽屉的“关联单据、下游单据、处理记录、凭证/NC”等区域作为入口。例如预存账户的退款申请、冻结申请、解冻申请、余额调整是当前账户自己的处理动作，应放列表；预收单已生成的退款单要去退款执行菜单处理，则默认放到详情的关联退款单入口。规则一句话：本页对象、本页抽屉、本页二级处理可以在列表；跳其他菜单处理下游对象，默认进详情关联入口。
核心业务场景的高频下游入口允许保留在列表操作列，不能为了形式上的“干净”强迫用户先进入详情页再操作。判断标准是：该入口是否是当前角色日常处理该对象的主要下一步、是否能直接进入对应业务对象列表或处理页面、是否不会在列表里完成复杂编辑。典型例子包括旅游线路列表进入团期/航次/班期/出行日期，团期列表进入执行或成本，订单列表进入收款/售后等高频承接入口；这些入口可以与编辑、详情并列放在操作列，但按钮数量仍需克制，低频动作放入“更多”。规则一句话：核心下游入口可放操作列，低频动作进更多，不能让用户为了高频操作绕进详情页。
所有业务列表页的操作列必须由“业务对象的主流程状态”决定，不能由“当前待办、异常提醒、风险提示、临期提醒”等提示性状态直接生成按钮。当前待办只用于解释对象为什么需要关注、当前卡在哪里、优先补什么，可以用于筛选、排序、标红和提示，但不是按钮生成器。设计列表操作列前必须先区分四类状态：主流程状态（如未开始、维护中、待提交、审批中、已完成、已终止，决定操作列）、当前待办（如资料待补、成本待复核、损耗待确认、临近 DL，只解释卡点）、异常/风险（如金额差异、库存不足、超期、毛利异常，只做风险提示）、下游承接状态（如已生成应付、付款中、NC 待推送，只展示下游结果）。例如成本结算列表中，结算状态决定操作列，当前待办只提示为什么不能提交或下一步要补什么；不能因为当前待办是“损耗待确认”就把按钮改成“确认损耗”，也不能因为当前待办是“结算依据待补”就把按钮改成“补依据”。按钮名称必须稳定、可预期，同一列表中不允许每一行因为待办不同而出现完全不同的操作词；具体缺什么、为什么不能提交、下一步补什么，应进入详情页或抽屉后展示。规则一句话：列表操作列由业务对象主流程状态驱动；当前待办只做提醒、筛选和优先级提示，不驱动按钮生成。
所有业务列表页的信息列必须展示“系统可判断、可核对、会影响业务流程推进的核心数据和数据缺口”，不能展示系统无法直接承接、无法数据化校验、无法形成业务对象状态的员工日常工作事项或沟通动作。列表里可以展示证件缺失人数、名单完整度、尾款未收单数、票号/舱号/铺位号缺失、交通确认数量、酒店确认号缺失、成本未确认、结算依据缺失、预付分摊未确认、NC 推送失败等数据结果或流程阻塞点；不展示“待联系供应商、待与地接确认、待通知客户、待安排领队、待催销售、下一步去确认、运营待处理”等人工作业提醒。若某个事项不能被系统以明确字段、数量、单据号、状态或缺口校验表达，只能作为工作台待办、消息通知、操作日志或详情页备注存在，不能进入一级列表主列。列表的“当前提醒/关键缺口”也必须表达缺少什么数据或哪个核心结果异常，而不是表达谁应该去做什么。规则一句话：列表展示流程数据缺口，不展示人的工作待办；人的动作进工作台、消息、日志或详情，不占主列表信息层级。
所有业务列表页和表格操作列禁止把员工沟通、催办、提醒、通知发送、客户/供应商跟进、执行记录、联系客户、联系供应商、安排人员等过程性工作作为按钮或更多菜单项。操作列只放三类动作：直接改变当前业务对象主状态的动作、进入当前业务对象详情或处理页的动作、处理系统可校验数据缺口的动作。催促、提醒、发送通知、保存跟进、记录沟通、查看执行记录等应进入工作台待办、消息通知、日志/记录、详情页或批量工具，不得占用列表操作列。若某动作确实会生成可追踪业务单据或改变主流程状态，按钮名必须使用业务结果词，例如“发起签署、生成通知单、确认回团”，不能使用“催、提醒、跟进、记录、联系”等员工动作词。规则一句话：列表操作列放业务状态动作，不放员工过程动作。
列表操作列按钮命名不能为了追求短而把不同业务阶段叫成同一个动作，尤其是财务链路中的申请、制单、付款、回单、核销、冲销、推送等必须区分清楚。按钮名称要先区分“去执行一个动作”和“查看某个执行结果/记录”：可操作状态用动词或动宾词，例如“申请、制单、付款、核销、冲销、推送”；已完成、已生成或只读反查入口必须使用“查看、详情、记录、回单、凭证”等查看语义，必要时允许超过两个字，例如“查看回单、查看冲销、查看付款”，不能把已完成记录继续叫“付款、核销、冲销”，也不能把“付款申请、付款单、付款执行、付款回单”都压成同一个“付款”。按钮两字优先是原则，不是压缩业务含义的理由；两个字会造成阶段混淆时，必须用四字以内的清晰名称。规则一句话：按钮短名要服从业务阶段，动词做动作，名词/查看词做记录入口。
“关键缺口/当前提醒”不是所有列表的标准列，不能为了套规则而强行加到每个列表。只有业务对象跨多个环节、状态链路较长、且用户确实需要在列表层面判断流程卡点时，才允许设置这类聚合列，例如团期管控、成本结算、交通采购履约、订单售后等复杂流程列表。普通主数据、配置、资源档案、供应商档案、线路列表、外采来源、单项服务、渠道配置等对象，如果主流程状态和少量核心字段已经能表达清楚，就不要额外增加“关键缺口”列；有风险或缺失时可在详情页、抽屉、筛选条件、状态标签、日志或消息待办里表达。设计前必须先判断该列表是否真的需要聚合缺口列：没有多个可系统校验的流程阻塞点，就不加。规则一句话：关键缺口只服务复杂流程列表，不是普通列表标配。
一般业务列表原则上不要同时展示两种或多种状态标签。一个列表的主状态永远只能有一个，它决定对象当前所处阶段，也决定操作列和主要筛选口径；次要状态、风险状态、来源状态、销售状态、下游承接状态等默认放到搜索区筛选条件、详情页、抽屉、日志或弱文本说明中，不要在列表中并列贴多个标签抢主状态。若列表已经用 Tab 承载主状态分组，例如全部/待发布/销售中/已停售，表格内原则上不再重复展示同一个主状态；此时允许在确有业务必要时再展示一个辅助状态标签，但不是必须，且只能用于补充判断，不能影响操作列。设计列表时必须先问：这个对象当前最核心的主状态是什么；如果回答不出来，就先重构状态口径，而不是继续增加标签。规则一句话：列表只有一个主状态；Tab 已承载主状态时，表格内最多按需补一个辅助状态。
后台页面和列表字段命名必须优先使用业务人员日常叫法，不使用偏系统建模、技术抽象或内部归集口径的词来做一级列表列名。能叫“付款对象”就不要叫“归集对象”，能叫“关联团期/项目团”就不要叫“关联对象”，能叫“供应商账单、成本范围、可申请金额、申请金额”就不要改成抽象的“对象、范围、金额口径”。只有在筛选、系统配置或技术说明中确实需要表达聚合规则时，才使用“归集、维度、对象”等抽象词。规则一句话：列名用业务听得懂的真实叫法，不用系统建模词。
后台业务列表的表格宽度默认按“首屏可判断、操作列可达、完整信息进详情”设计：后台业务列表默认支持横向滚动，滚动只发生在表格容器内，不撑破页面外壳；操作列必须固定在右侧，不能让用户横向滚动很久才看到操作按钮；每个单元格最多承载三层内容（主信息、辅助信息、一个关键标签或补充），超过三层应拆到详情页、抽屉、日志或二级页面；列表只展示当前场景下用于识别对象、判断状态、发现异常和完成处理的关键字段，不为了“完整”把低频字段、解释性内容或全量详情堆进表格。规则一句话：后台业务列表默认可横向滚动，操作列固定右侧，单元格最多三层，完整信息进详情或抽屉。
后台业务列表的固定操作列必须使用明确的固定宽度，并且不能参与表格剩余宽度分配。使用 `table-layout: fixed`、`width: 100%` 或表格容器宽于列宽合计时，必须指定某一业务信息列作为弹性吸收列，或让表格按内容最小宽度横向滚动；不能让 sticky 操作列被浏览器撑宽，导致操作按钮右侧出现大片空白。操作列的 `col`、`th`、`td` 应保持一致的 `width/min-width/max-width`，按钮区域左对齐且宽度只服务实际操作数量。规则一句话：表格多余宽度给业务信息列，不能给固定操作列。
后台业务列表的“列数”默认指业务信息列，不包含复选框、展开箭头、拖拽手柄、序号等窄工具列。批量选择复选框可以作为独立窄列存在，不应为了凑业务列数强行塞进主识别列，导致对象名称、日期、人数等核心识别信息被挤压或看起来合并。工具列必须固定窄宽，业务主识别列必须保留足够宽度和清晰表头。规则一句话：工具列不算业务列，不能挤占主识别列。
后台业务列表做列宽和单元格排版时，必须区分“关键短值”和“可省略长文本”。数量、金额、百分比、状态标签、订单数、缺口数、业务流转编号、操作按钮等用于判断和处理的短值必须完整显示，不能因为单元格内部 flex/grid 收缩、整格 overflow hidden、进度条占满宽度、固定列宽不足或按钮/链接负边距而被裁切；省略号只允许用于线路名、客户名、供应商名、长备注、长说明等可通过详情补看的长文本。列表内的按钮和链接若放在 overflow hidden 的容器中，不允许使用负 margin 扩大点击区导致文字被父级裁边。验收列表时必须检查表头宽度、列宽合计、单元格内部收缩规则、按钮链接边界和小屏横向滚动后的显示结果。规则一句话：列表可以横向滚动，但关键短值不能被省略或裁切。
后台业务列表里的数量型跳转入口必须只让数字本身成为蓝色可点击元素，单位、对象名和后续状态说明保持普通正文或弱文本颜色。例如“128条线路”“36个团期”“4个服务项目”“44单”这类入口，应写成数字链接/按钮 + 普通单位文本，不能把“条线路、个团期、单”等整段都包进蓝色链接。规则一句话：数量入口只蓝数字，单位和说明不蓝。
后台业务列表单元格的多行内容必须用语义容器或块级文本行统一组织，例如主信息、辅助信息、状态标签各自作为明确的块级/行内块元素，并由 CSS 控制固定间距。禁止在列表单元格里同时使用 `<br>` 和 `display:block` 的 `span/strong/a` 等元素来制造两行内容，因为这会额外插入一整行空白，导致两行内容中间距离异常。若某个单元格需要两行或三行，直接输出连续的语义元素，由 CSS 设置 `margin-top: 3px` 等标准间距；只有纯文本段落或详情说明中确需文本换行时才允许使用 `<br>`。规则一句话：列表多行内容不用 `<br>` 叠块级元素，换行和间距交给语义 class。
后台业务一级列表的单元格默认上下垂直居中，尤其是状态、负责人、操作、金额、关键缺口等判断和处理列，必须与同一行的多行主信息保持视觉居中；不能为了让某个多行单元格好排版，在整张一级列表上写 `vertical-align: top`。多行内容应在单元格内部用语义容器组织，确需顶部对齐的只限明细表、费用拆分表、日志表等局部台账型表格，并用语义 class 单独控制。规则一句话：一级业务列表默认垂直居中，台账明细才局部顶对齐。
并列对照型字段使用进度条时，例如应收/实收与成本/应付、库存/已售与名单/确认等，必须使用同一套进度条组件、同一列宽口径和同一进度条宽度，不允许一列带额外文字、一列不带，或被旧的 `td:nth-child(n)` 列序样式影响后呈现不同长度。新表格样式优先用语义 class 精确控制，避免依赖脆弱的列序选择器。规则一句话：对照型进度条必须等宽、同结构、同组件。
后续优化任何列表页时，必须同时做字段取舍审计，不能只围绕旧表格已有字段压缩排版。先判断当前角色在列表层面需要用哪些字段完成识别、筛选、排序、状态判断和高频处理；旧列表里低频、装饰性、内部口径、详情型、统计看板型、人工待办型字段要主动移除或降级到详情/抽屉/筛选/日志；若缺少影响当前场景判断和处理的关键字段，也要主动补入列表。字段是否展示不以“旧页面有”或“数据能取到”为准，而以当前业务对象、角色任务、状态流转和下游影响为准。规则一句话：列表优化必须删非必要字段，也要补必要缺口字段，不能只压缩旧字段。
后台业务列表原则上不展示系统内部编码或主数据编码，除非该编码是业务人员日常核对、对账、收付、结算、审批或跨系统沟通必须使用的业务流转单据号。线路编号、资源编号、供应商内部编号、配置编号、模板编号等默认不进入一级列表；订单号、团期编号、航次/班期编号、售后单号等履约和客户沟通相关编号可以展示；财务模块、资金流水、认款、收款、退款、应收、应付、付款、回单、对账、结算、发票、NC、资金池等和钱相关的编号必须优先保留，例如流水号、认款单号、收款单号、退款单号、应收应付单号、付款申请号、回单号、对账单号、结算单号、发票号、NC 凭证号等，因为财务核对需要精确匹配，不能只靠名称或状态判断。规则一句话：列表不露内部编码；但履约核对编号和所有钱相关编号必须保留展示。
交通采购、锁位、包舱、包列等动态资源列表展示路线时，应优先使用业务人员扫表能快速判断的城市/航线短名，例如“北京-巴黎 / 巴黎-北京”，不要把机场名、航站楼、码头、站点全量塞进一级列表主信息，也不要在路线行后拼接航司、船公司、运营商或供应商名称；完整机场、港口、站点、供应商和航班/舱位明细进入详情或抽屉。此类列表的“关联团期/航次/班期”列默认展示数量入口，例如“2个团期”，点击后带入筛选条件进入团期管控列表；不要在资源履约列表里直接铺开多个团期编号，除非当前页面任务就是精确核对某个团期编号。规则一句话：动态资源列表用城市短名识别路线，路线行不拼供应商，关联团期显示数量入口并带条件跳转。
付款申请、应付管理等付款发起类列表必须区分“应付金额、可申请金额、本次可申请、本次申请金额”。应付金额是成本或账单总额；可申请金额是列表层面的余额/当前可进入申请的金额，必须作为付款申请列表的核心字段展示；本次可申请是进入申请抽屉后，受所选成本项、付款节点、合同/账单依据和审批规则限制的本次上限；本次申请金额是用户实际提交的金额。不能用应付金额冒充可申请金额，也不要把本次可申请放到一级列表里制造列宽压力。规则一句话：付款申请列表看可申请金额，抽屉里处理本次可申请和本次申请金额。
功能设计要优先服务真实操作效率：列表页突出待处理、临期、异常、可操作状态；详情页串起上下游单据和下一步动作；抽屉用于快速处理当前对象；弹窗只做打断式确认；批量动作只用于行业里真实会批量处理的场景。若旧流程让角色频繁跨模块、重复录入、看不到关键状态、需要理解内部建模语言，必须重构流程和页面结构。
任何页面改造都不能只追求“数据逻辑能通”，还要符合旅游行业的实际协作方式：产品经理维护线路和方案，计调/团控处理团期、资源占用和执行，销售顾问处理订单和客户，票务/资源调度处理航班舱房铺位等动态供给，财务处理收付结算发票NC，审批人处理跨模块审批。角色边界不清时，先重构边界再改 UI。

### 抽屉信息层级与文案规则
后续所有右抽屉，以及承载业务内容的弹窗，必须按“标题区只识别动作；内容区用摘要卡或轻量对象识别条识别业务对象；字段区承载业务判断；底部只完成操作”的信息层级设计。标题区只允许放当前动作标题和关闭按钮，例如“新增会员”“团期订单概览”“应付差异处理 - AP20260623003”；不放模块路径、类型小标题、副标题、说明文案、状态标签、上下游摘要或业务内容。团期编号、出发日期、订单号、供应商、金额、主状态等业务信息必须进入内容区，不得堆在标题下。
抽屉和业务弹窗默认不展示解释性文案。凡是“用于……”“这里只……”“当前抽屉不……”“保存后可……”“本抽屉只确认……”这类教用户理解页面用途、流程边界或系统设计的标题说明、分组说明、表单说明、表格说明、底部提示和弹窗说明，都必须删除或改成真实业务字段、状态、风险标签、操作结果 toast。分区标题只写业务名词，例如“基础信息”“证件与偏好”“重点订单”“差异依据”“本次处理”，不在标题下补解释句。所有抽屉分区标题必须统一使用左侧蓝色竖线，优先通过 `drawer-section-head` + `drawer-section-title` 的全局样式实现；历史抽屉仍使用 `form-section-title` 时，也必须由抽屉全局样式统一补齐蓝色竖线。不允许每个页面各自画一套标题样式。
业务上下文不是说明文案，必须保留在内容区摘要卡或轻量对象识别条中。若当前判断对象需要通过供应商、费用项目、关联线路/团期/班期、出发日期、客户/渠道等信息识别，应放在摘要卡对象标题下或轻量识别条中作为辅助业务上下文，例如“丝路专列运营商 / 铺位包列 / 丝路专列深度游 / 丝路经典线 / TR-SILK-20250815-001”。这类上下文只能表达当前处理判断所需的高价值业务信息，不能写成页面说明；金额、数量、占用、差异、待收、状态、缺口等关键短值若是当前处理判断重点，必须从长句中拆出作为摘要字段独立展示，方便扫读和核对。
抽屉结构可以优化，但字段名、分区名和按钮名必须优先沿用原页面已有且业务人员熟悉的叫法。不能为了套用新结构或追求“更规范”而随意重命名，例如把原本清楚的“产品类型”“我方销售政策”改成偏系统化或含义漂移的“入库方式”“入库设置”。只有原名称明显不准确、过度技术化、属于内部建模词或会误导业务理解时，才允许改名；改名必须改成旅游业务人员日常会说、能直接判断含义的词，并确保新名称比原名称更准确、更业务化。规则一句话：结构可重组，业务字段名先沿用；要改名必须有业务必要，且越改越像业务人员的话。
后续分析或优化任何页面的抽屉时，必须先做“抽屉业务结构审计”，不能默认沿着旧抽屉补字段，也不能为了显得彻底而默认推翻旧结构。先判断当前页面的真实使用角色、当前抽屉对应的业务动作、操作对象、上游依据、下游结果和状态变化是否成立；旧结构合理就保留并精修字段，旧结构边界不清、动作重复、弹窗/抽屉拆分不符合操作习惯、底部按钮与处理结果不一致、或字段只是抽象兜底时，才做局部调整；如果多个分区、多个抽屉或操作路径都建立在错误业务逻辑上，必须提出并执行必要重构。每个抽屉字段都要逐项判断是否当前操作必需、是只读核对还是可编辑输入、是输入框/下拉/搜索选择/日期/金额/上传/只读结果、是否需要随处理结论或业务对象联动显示、填写后改变哪个状态或生成哪个下游记录。多种处理结论属于同一业务对象和同一处理动作时，原则上在同一个处理抽屉内通过“处理结论/本次处理”联动字段和底部主按钮完成；只有确实是独立业务对象、打断式确认或跨页面流程时，才拆成单独弹窗或跳转。规则一句话：先判断旧抽屉业务结构是否成立，成立则精修，不成立才调整或重构；字段、联动、按钮和状态结果必须细到真实可用。
所有抽屉必须先按业务任务分型，再决定摘要、正文、按钮和宽度，不能用一个通用模板硬套所有场景。全局抽屉分为七类：
一、创建/编辑/配置类，用于新增、编辑、维护、配置等以字段填写为主的操作，例如新增会员、编辑团期、维护资源、配置规则。内容区直接进入表单，不默认放摘要卡；若编辑已有复杂对象需要先识别对象，才在表单前放轻量对象摘要。字段顺序按“必填识别信息、影响状态或下游的字段、补充信息”组织，分组只保留业务标题，不写分组解释。底部主按钮使用“保存、确认保存、提交”等结果词。
二、详情/速览/记录类，用于订单详情、团期订单概览、日志、版本、台账、付款详情、主数据档案详情等快速判断场景。订单、团期、单据、对账、付款等业务处理或核对对象，主体应先用对象摘要卡回答“正在看哪个对象、当前主状态是什么、关键金额/数量是什么”，再承载明细表、记录、日志或时间线。分销渠道、供应商档案、资源档案、配置档案等主数据档案型详情，如果摘要字段会与首个 Tab 或各 Tab 内容高度重复，不使用摘要卡，只在内容区顶部放轻量对象识别条或直接进入 Tab；识别条只放对象名称、必要状态、类型/区域/负责人/电话等少量识别字段。底部只保留“关闭、进入详情页、进入处理页面”等少量入口，不在速览抽屉里做复杂编辑。
三、业务处理/状态推进类，用于资源确认、名单处理、发起售后、发送通知、调整库存价格、申请付款、安排领队等会改变业务状态的动作。结构为“对象摘要 + 判断依据 + 本次处理项”。摘要卡放当前处理最需要判断的团期、线路、出发日期、占用数、待确认数、主状态等；判断依据放为什么能处理、缺什么、影响什么；本次处理项才放输入字段。底部主按钮必须使用明确业务动作词，例如“确认资源、提交申请、发送通知、确认调整”。
四、核对/对账/差异处理类，用于应付对账、差异处理、认款复核、付款异常、收票、结算确认、供应商账单确认等需要对照两套或多套数据后作出判断的场景。摘要卡必须突出核对结论和核心差异，例如单据号、供应商/费用项目/关联业务、系统金额、对方账单金额、差异金额、当前状态。正文第一块放核对依据和完整字段，第二块放本次处理或确认结果。摘要和正文允许少量字段重复，但摘要只放判断重点，正文放完整依据，不能两块都做成同层级“基础信息”。
五、选择/筛选/关联/分配类，用于高级筛选、选择目标对象、选择代理入库、分配到团期、销售授权、指定分销商等。结构为“来源对象识别信息（按需）+ 搜索/筛选 + 可选列表/本次选择项”。这类抽屉重点是快速找到、勾选、确认，不承载复杂处理；如果只是给某个对象配置授权范围或关联范围，顶部最多展示分销渠道、供应商、团期、资源等来源对象的轻量识别信息，不强制使用摘要卡，更不能把尚未形成的授权范围、分配结果、佣金结算、账户状态等当作本次结果提前放进摘要。来源对象识别信息必须使用轻量识别条或紧凑对象行：第一行显示对象名称和必要状态，第二行用弱文本横向展示类型、区域、负责人、联系电话等少量识别字段；禁止做成带粗边框的大表格、description-grid 详情表、指标卡或摘要卡。选择/关联类抽屉不新增“已选结果”“提交摘要”“结果汇总”这类独立汇总分区；已选数量、分配数量、校验结果、影响范围等信息应在可选列表行内、本次选择字段、主按钮状态或操作后的 toast/状态变化中表达。底部主按钮使用“确认选择、确认分配、确认授权”等动作词。
六、批量/导入/上传/AI辅助类，用于批量维护出行日期、导入银行流水、上传文档、解析结果、AI定价建议、AI智能建品等。结构按当前任务必要内容组织，常见为“来源输入 + 解析/批量预览 + 异常结果 + 应用确认”，不能为了套结构强行补齐所有分区。摘要卡只放批量范围、影响数量和关键风险；必须表达失败、异常、可忽略项或需人工确认项。解析设置、导入设置、AI 参数等配置项只有在用户本次必须做业务决策时才展示；若入口、文档类型或系统默认规则已经能确定处理方式，不得额外展示“解析对象、低置信度处理、处理结果”等设置项。AI 和导入结果不得直接替代业务判断，底部主按钮使用“确认导入、应用结果、确认批量维护”等。只有当预览区能展示基于当前上传文件、勾选对象或解析结果生成的动态内容时，才允许展示解析预览/任务预览，例如字段数、异常数、低置信度字段清单、影响线路数；若只是“生成任务、当前上传文档、处理完成后怎样”这类固定结果说明，不得放入内容区，应通过按钮文案、toast 或状态变化表达。
七、复杂创建/长流程类，用于新建线路、新建团期、新建订单、创建独立成团、新建定制需求、开排航次等跨多个业务对象或会影响产品、团期、库存、订单、客户、成本、履约的流程。此类不强制套普通抽屉，可使用超大抽屉、步骤条或完整页面；内容按真实业务流程分段，而不是按数据库字段堆叠。未被明确要求重构时，新建线路、新建团期、新建订单保持独立，不借全局抽屉调整顺手改动；一旦专门重构，也必须遵守标题干净、说明文案删除、业务上下文进内容区、底部只放操作按钮的规则。
摘要卡和正文必须避免“平级重复”。摘要卡只放当前角色做判断最需要扫读的高价值字段，例如对象编号、业务上下文、主状态、关键金额、关键数量、差异、缺口、风险；正文放完整依据、可编辑字段、明细表、记录或本次处理项。若某个字段已经在摘要卡出现，正文中再次出现时必须承担不同用途，例如用于完整核对、表格明细、可编辑输入或留痕，不得简单复制一遍占版面。
抽屉内容区必须按自然内容高度排列，不能因为抽屉 body 占满剩余高度而把业务分区拉开。`.modal-body` 若使用 `display:grid`、`display:flex` 或类似布局，必须明确 `align-content:start`、`align-items:start` 或让子分区保持自然高度；普通表单抽屉优先使用自然文档流，不在满高滚动容器上用会 stretch 的 grid 行布局。分区之间只用标准间距和分隔线表达层级，禁止出现大段无业务含义的空白。
抽屉内表格必须先按业务必要性压缩列，再按抽屉正文宽度设计列宽，不能把列表页表格思路直接搬进抽屉。普通右抽屉内的 2-5 列处理表、价格矩阵、选择明细、核对明细，默认不使用横向滚动，表格容器宽度必须为 `100%`，表格必须铺满抽屉正文宽度并使用 `table-layout: fixed`，通过 `colgroup` 或语义列 class 给每一列设置明确宽度；列宽合计必须扣除抽屉正文左右 padding、表格边框和单元格 padding 后仍能完整容纳。输入框、选择器、按钮等控件宽度必须小于所在列的可用内容宽度，并设置 `max-width: 100%`、`min-width: 0`、`box-sizing: border-box`，不能让控件默认宽度反向撑破列，也不能用 `fit-content` 或过窄固定表格造成右侧大片空白。规则一句话：抽屉内短表格默认在原抽屉宽度内铺满、列宽固定、控件随列压缩，不截断也不留大空白。
抽屉内表格只有在业务上确实需要 6 列以上、长文本不可删减、金额/差异/状态/操作都必须同屏核对时，才允许使用更宽抽屉或表格横向滚动；使用前必须先删掉低频列、把长说明移到详情/备注/日志、把可合并字段合并成主辅两行。确需加宽时只加宽抽屉，不把表格缩成窄块；确需横向滚动时滚动只能发生在表格容器内，不能让抽屉 body 或页面横向滚动，关键短值、状态、输入框、操作按钮必须完整可见。验收时必须同时检查表头、每列单元格、输入框、select 箭头、按钮边界、表格右边界和抽屉正文右边界：不能裁切、不能重叠、不能出现无业务意义的大空白。
抽屉底部只放操作按钮，例如“取消 / 关闭 / 保存 / 确认处理 / 进入销售中心”。底部不放流程说明、影响说明、免责提示或页面用途说明；必要风险在内容区用业务字段或状态标签表达，操作结果通过 toast 或状态变化表达。新建线路、新建团期、新建订单等接近完整业务流程的复杂创建入口，未被明确要求重构时不要借全局抽屉调整顺手改动；一旦专门重构，其标题区、说明文案、业务上下文和底部区域也必须遵守本规则。

### 改动半径与验收范围规则
后续模块细节优化必须按“改动半径”确定验收范围，不允许默认把每次页面改动都扩大成订单、收款、执行、成本、结算、付款、发票、NC 的全流程回归。只有改动主数据链路、跨模块对象关系、状态机、编号规则、共享脚本、导航归属、存储结构或会真实影响下游单据生成的字段时，才做跨模块链路回归。
资源模块改动只验资源对象本身、维护角色、入口归属、产品页引用/直接填写/沉淀是否合理，不默认验证订单和财务；产品模块改动只验线路、方案、素材、费用政策、排期入口和团期/航次/班期/出行日期承接，不默认验证收付款和NC；履约中心改动验排期列表、详情、执行入口、成本入口和三组状态机，只有影响订单锁名额或售后时才继续验订单；销售中心改动验下单、名额锁定、改价、售后、应收、退款事件；财务中心改动才验应收、认款、应付、付款、结算、发票、NC。
执行页面改造时必须先判断本次改动真实影响到哪些上游和下游，验收只覆盖这些对象及其直接承接页面。不能为了“完整闭环”反复扫描无关模块，也不能因为后续链路存在就把当前模块优化拖成全系统回归；但如果发现当前改动破坏了主链路字段、对象归属或状态流转，必须主动扩大验收范围并说明原因。

关键归属规则：
- 资源主数据属于资源模块；基础资源、航空资源、邮轮资源、专列资源、供应商、领队资源都挂在资源下。
- 供应商属于资源主数据；分销商、代理商属于渠道对象，不能复用供应商页面或挂到资源下。
- 产品市场属于 产品中心 > 外采产品，只展示可外采、可代理的线路产品，不放自营产品；确认代理入库后进入代理产品。
- 代理产品属于 产品中心 > 外采产品，承接产品市场选择后的代理入库、包装、价格、配额和后续销售准备。
- 产品管理属于 产品中心 > 自营产品，是自营产品统一列表；参团游、邮轮、专列、自由行、研学、单项服务等作为 Tab 或产品类型在页面内维护，不再单独作为左侧菜单。
- 单团项目属于 产品中心 > 自营产品；它承载 MICE/定制团等项目型成团流程，不是标准线路库，也不和 MICE 项目另建重复菜单。
- 团期管控属于 履约中心，是团期、航次、班期、项目团的核心入口；日历视图、批量开排、新建团期放在团期管控页面顶部操作区，不作为左侧菜单。
- 团期结算属于 履约中心，统一承载团期成本、供应商费用和付款申请；业务侧确认成本和付款申请，财务侧承接审批后的应付、付款执行、回单和NC。
- 签证办理属于 履约中心，承接订单游客材料、送签、出签和异常跟进，不替代资源模块的签证国家配置。
- 交通采购属于 履约中心 > 交通采购，承接机票切位、邮轮包舱、专列包铺等带日期、数量、成本和付款节点的动态供给，不放在资源模块日常操作。
- 出团执行、名单管理、证件资料、签证进度、出团通知、回团结算等不单独挂左侧菜单；从团期详情、团期管控、团期结算或对应行操作承接。
- 资源模块只维护可复用基础资源。基础资源维护景点、酒店、餐厅、车型、别名等低频基础资料，只保留名称、所在地、分类和基础属性；基础资源不走审批，列表和维护抽屉不展示状态列或审批状态；不在基础资源页展示适用线路、引用线路/团期、门票、淡旺季价、可用周期、供应商报价、来源系统、匹配规则、影响模块等下游业务或系统内部信息。邮轮资源维护船公司、船只档案、船上设施、甲板图、基础舱型、邮轮模板、母港/停靠港；专列资源维护运营商、专列模板、车厢/席别/铺位基础结构、起终点、途经站；自由行相关资源维护酒店、航班/锁位、车型、景点、供应商、接送机服务和当地玩乐资源。资源里的模板不能等同于某个可售线路/方案的售卖结构。
- 航空资源下的航线库只维护基础航线模板，例如出发城市/机场、到达城市/机场、中转城市、航司、常用航班、航段类型、飞行时长和适用目的地；航线库不维护供应商、采购方式、协议规则、退改规则、结算口径、成本科目或具体锁位。航司协议只维护航司/供应商合作规则，例如协议周期、适用航线范围、采购方式、舱位规则、定金规则、释放规则、出票规则、退改规则和结算币种/结算方式；具体团期锁位、名单确认、释放、出票和票号回写属于履约中心的团期管控或团期详情，不作为资源模块日常操作入口。
- 资源模块不是业务数据看板，也不是风险、绩效、财务反查入口。资源列表、详情、抽屉和弹窗只展示维护档案所需的核心基础字段，不展示近期派团、评分、合作风险、冲突风险、业务量、销售额、引用线路、引用团期、使用次数、待对账、应付、付款、NC、财务单据、经营分析等下游业务数据；这些数据后续统一进入数据模块、团期详情、订单详情或财务模块查看。
- 资源模块状态必须按对象风险判断：景点、酒店、餐厅、车型、航线库、船只档案、邮轮模板、专列模板等低风险基础资料默认不走审批，一级列表不放状态列；如需要停用，只做简单启用/停用开关并放在维护抽屉或行操作里。供应商、领队、航司协议、合同资质等准入/合规对象可以保留合作状态、准入状态或协议状态，但状态必须服务准入和可用性判断，不承载风险评级、绩效评分或财务处理状态。
- 资源、模板、协议等配置类页面默认以业务名称作为主识别字段，不让员工手工填写内部编号，也不在一级列表展示系统编号。系统如需编号应自动生成并用于后台关联；只有订单号、团期编号、航次编号、班期编号、服务单号、结算单号、付款申请号、NC凭证号等业务流转单据号，才在生成后展示给用户核对。
- 方案层维护具体售卖组合结构：邮轮方案维护本线路采用哪些舱型参与售卖，专列方案维护本线路采用哪些车厢/铺位结构，自由行方案维护机票+酒店+接送机等套餐组合；团期、出行日期或套餐批次维护具体日期、价格、库存和截止节点。
- 转团、改签改期、退款申请属于销售中心售后；退款执行属于财务。
- 门店是自有销售网络和资金节点，归入门店中心；分销商、代理商、OTA 属于渠道中心。
- OTA平台属于渠道中心；OTA订单/退款可作为销售中心反查或售后承接页面，但渠道日常配置、商品映射、平台对账和异常日志归渠道中心 > OTA平台。
- 财务口径报表属于财务 > 财务报表。
- 审批中心独立承接线路发布、价格调整、渠道授权、成本、付款、退款、折扣、转团、结算等跨模块审批。

字段分层规则：
- 跟团、邮轮、专列统一采用“线路 → 方案 → 团期”三层模型；字段标准以 docs/route-plan-schedule-field-standard.md 为准。
- 线路层只维护长期稳定的售卖信息、通用费用政策和素材，不维护具体出发日期、当期价格和当期库存。
- 方案层维护具体行程和服务标准；邮轮方案必须维护舱型结构，专列方案必须维护车厢/铺位结构。
- 团期层维护日期、状态、价格、库存、截止节点和执行确认；邮轮团期按舱型维护价格库存矩阵，专列团期按车厢/铺位维护价格库存矩阵。

## 三大核心原则（每个页面设计必须体现）

### 原则一：业财一体
凡涉及收款、退款、付款、改价操作完成后，页面显示绿色"财务联动"区块，内容是自动生成的财务单据编号和状态。例如：✅ 已自动生成应收单 SK20240618001，状态：待审核。

### 原则二：AI原生
以下页面必须有AI功能入口：
- 线路创建页：AI一键生成行程内容按钮
- 订单详情页：AI风险提示（如证件即将过期、出发日临近未收款等）
- 意向订单页：AI跟进话术建议
- 团期成本页：AI毛利预测
- 客户详情页：AI复购预测和标签建议
AI入口用紫色渐变按钮标识，标注"🤖 AI"字样。

### 原则三：数据驱动
所有报表统计类页面底部必须有"数据口径说明"灰色区块，注明：统计口径、数据更新频率、数据范围。

## 全局视觉风格规范
整体页面必须保持简洁、干净、克制，优先通过清晰的信息层级、留白、边框、状态标签和按钮状态表达结构。
不要为了装饰而给公司名、供应商名、线路名、部门名、工作空间名前添加首字头像、字母Logo、色块Logo或无业务含义的图标。
客户、联系人、员工等真实人物只有在列表需要快速识别身份时才可以使用头像；若无明确头像素材，优先使用姓名文本，不默认生成首字头像。
业务对象列表（订单、线路、公司、供应商、部门、工作空间、团期等）默认使用文字、tag、表格列、选中态和操作按钮呈现，不使用拟头像风格。
视觉设计避免模板化、花哨化，避免过度装饰、过多色块和无意义图标，后台页面以清爽、专业、高效为主。

## 后台UI细节规范（参考 Ant Design Vue）
后台页面默认采用 14px 基准字号和 22px 行高；辅助说明、表格次要信息、标签文字可用 12-13px，但同一组件内必须统一。
页面主标题在后台一级列表页和二级详情/新建/编辑页默认使用 18px/26px、600 字重；只有工作台、看板或需要视觉强调的总览页才可使用 20px/28px、600-700 字重。卡片标题使用 14-16px/22-24px；不要在紧凑后台页面使用过大的展示型标题。
顶部页面标题区参考 Ant Design PageHeader：内容卡片内边距 16-20px；标题左侧返回入口必须和标题基线对齐。
返回入口使用 14px 文字和 14px 图标尺寸，颜色用主色，不使用过大的文本符号；二级详情/新建/编辑页返回入口文案统一只显示"返回"，不拼接上级页面名称，上级归属由顶部位置提示和左侧菜单表达。
后台页面内容区默认不放面包屑；一级列表页只显示页面标题和操作区，不展示"模块 > 当前页"面包屑，也不在标题区放"共N个"这类数据汇总；后台页面默认不放解释性副标题、说明文字、`page-subtitle`、蓝色信息框、说明条、单独占一整行的业务说明或"页面说明文字"（如"佣金规则与渠道类型绑定..."、"单项服务：..."、"自由行线路：..."等）；标题和主操作必须在同一个紧凑标题区内，不为了标题单独放一个空白卡片或再额外放一块说明卡片；即使页面需求中单独写了"说明文字/页面说明/页面顶部说明文字/面包屑：模块 > 当前页"，也必须以本规则为准，不在内容区实现该面包屑或说明文字；二级详情/新建/编辑页优先使用"返回 + 当前标题"，不再重复展示"模块 > 列表 > 当前页"面包屑。
后台页面禁止把产品建模、字段分层、数据链路、对象链路、业财流转逻辑做成说明卡片、说明表格、流程解释区或大段文字放进页面内容；页面只能呈现业务用户需要操作或核对的真实数据、状态、单据和动作。类似"业务锚点/库存锚点/对象链路/数据链路/字段口径/层级说明/成本来源于..."这类内部建模语言不得出现在页面 UI 中。
页面字段命名必须使用业务实际叫法，不使用抽象技术名或内部建模名。优先使用：线路类型、团期编号、航次编号、班期编号、服务单号、订单号、成本单号、费用单号、应付单号、付款申请号、结算单号、NC凭证号。不要使用"库存编号""库存锚点""业务锚点"等不通用名称；自由行相关页面优先使用"出行日期""套餐批次""采购批次""供应商确认号"等业务叫法，避免把面向用户的字段命名为"日期库存"。
一级列表页禁止在标题区、筛选区上方或表格上方放置大号 KPI 统计卡片、数据汇总卡片、轻量汇总条、蓝色汇总条或 `.kpi-grid/.kpi-card` 汇总区（如 在售数量、供应商数、本月销售额、利用率、本月收入、本月服务费收入 等）；即使页面需求写了"顶部统计卡片/统计卡片横排/核心指标卡片/顶部统计/摘要"，也必须忽略，不做大卡片或轻量条汇总。KPI 卡仅用于工作台、报表页、详情总览、分析看板等非一级列表场景。
一级列表页只展示角色扫表时必须用于识别对象、判断状态、排序筛选、发现异常和进入下一步动作的高频字段；非关键核对信息、低频说明、资源来源、模板来源、字段来源、产品内调整、内部匹配状态、内部同步状态、辅助性说明文字等默认不进列表列，也不为了让列表"看起来完整"而增加列宽。这类内部信息也不要作为说明字段、说明卡片、状态标签或详情项显示在详情页、编辑页、抽屉、弹窗或二级页面中；除非它本身就是当前角色必须操作的表单控件或审批判断依据，否则不要出现在页面 UI。
顶部导航栏的位置提示必须与当前页面归属一致；顶部位置提示从业务模块开始展示，不显示"商户端/运营平台"这类系统端根节点；不在左侧菜单中的二级/三级页面必须在导航脚本中配置归属菜单和当前页标题，例如 product-self-edit.html 归属 产品中心 / 自营产品 / 产品管理，绝不能因未匹配菜单而回退显示 工作台。
商户端顶部右侧组织/部门切换入口必须由共享导航脚本统一提供，不在单页内重复实现；点击当前组织名称打开下拉树，数据层级支持 集团 > 子公司 > 中心 > 门店/部门，但下拉展示默认隐藏集团根节点、从子公司开始展示；顶部提供搜索框，节点类型用轻量文字标签和颜色区分（子公司、中心、门店、部门），节点名称后不显示组织编码，选中节点后更新顶部当前组织名称。
仅三级及以上层级、跨模块多入口页面、或需要展示业务归属时才使用内容区面包屑；面包屑使用 14px/22px，分隔符统一为 >，可点击节点用主色，当前节点用普通弱色或正文色，不加粗到抢标题层级。
按钮参考 Ant Design 默认尺寸：高度 32px，字号 14px，左右内边距 15px；同一操作区按钮高度、字号、圆角必须一致，按钮间距 8-12px。
一级列表页右侧操作区如存在多个同类批量动作，不要平铺多个"批量XX"按钮；应合并为一个"批量操作"下拉按钮，菜单项放具体动作（如 批量上架、批量下架、批量导出、批量分配 等）。"批量操作"必须使用 .btn.btn-secondary 灰色边框按钮样式，不能做成表格行内"更多"那种纯文字按钮，避免与"新建/提交/确认"等主操作争抢视觉层级。
常规图标尺寸统一为 14px 或 16px；按钮内图标默认 14px，导航/菜单图标默认 16px；同一区域不要混用 12px、18px、20px 图标。
后台布局间距使用 4px 基础倍数：小间距 8px，中间距 16px，大间距 24px；后台列表、表格和工作台内容默认采用更紧凑的 16px 内容区留白，只有详情总览、看板或需要强调的区块才使用 24px 大留白；避免 10px、18px、22px 等零散间距连续出现。
表格、筛选区、弹窗等组件优先对齐 Ant Design 的紧凑后台密度：输入框和选择器高度 32px，表格行高保持可扫读，不为了装饰额外撑高。
后台弹层默认遵循"业务操作用右抽屉、打断确认用居中小弹窗"：新建、编辑、配置、选择类型、批量维护等承载较多表单/卡片内容的弹层使用右抽屉结构 .modal-overlay.drawer-overlay > .modal.drawer-modal；右抽屉必须从屏幕右侧向左平滑滑入，使用共享 .drawer-overlay.show .drawer-modal 的 drawer-slide-in 动画兜底，不能只依赖 display:none 后的 transition，避免突然跳出；关闭时如需滑出，JS 应先移除 show 并延迟约 240ms 后再隐藏/清理状态；删除确认、复制确认、上下架确认、批量确认、未选择提示等打断式确认使用居中 .modal.modal-sm.modal-confirm，样式保持简洁克制，不使用过大的标题、按钮或大面积分隔线。
一级列表页筛选区应作为轻量工具条直接承载筛选项，不使用额外卡片背景、粗边框或阴影；只保留必要的输入框边框和页面区块留白。筛选工具条必须区分"筛选动作"和"页面主操作"：搜索/查询/重置属于筛选动作，必须紧跟全部筛选条件之后；新建/新增/批量/导出/同步/AI智能上架等属于页面主操作，必须固定在筛选工具条右侧或标题右侧的主操作区，不能混在筛选条件流里。筛选条件较多换行时，搜索/查询/重置跟随筛选条件流出现在最后一行条件之后，右侧页面主操作仍保持右上角对齐。
筛选项默认固定业务宽度，不按整行剩余空间拉伸，不为了铺满容器而把单个输入框拉长；条件较少时保持自然宽度，条件较多时自动换行。
筛选项宽度标准：普通文本输入使用约220px；小型下拉使用约136px；日期范围使用约318px，内部两个日期输入各约136px，中间连接文字保持紧凑。
一级列表页筛选区推荐使用 .filter-card.filter-card-compact；普通条件使用 .filter-item；短下拉使用 .filter-item-sm；日期范围使用 .filter-item-wide 或 .filter-item-date-range；状态下拉可使用 .filter-item-status。
筛选区 select 必须使用统一下拉箭头样式，箭头距离右侧约10px，不使用浏览器原生箭头导致的贴边或偏移效果；小型下拉需要预留右侧内边距，避免文字和箭头重叠。
移动端筛选项才允许 100% 宽度纵向排列；日期范围在移动端占满一行，内部两个日期输入平分可用宽度，不横向溢出。
筛选区操作按钮使用次级按钮样式，搜索/查询/重置在同一视觉层级，不使用黑色或强主色填充按钮；导出、同步、批量操作、新建/新增等不属于筛选动作，应放入页面主操作区。主色按钮只用于新建、提交、确认等明确主操作。
左侧导航选中态必须停留在当前最终菜单项；有子菜单时父级只负责展开，选中态优先用背景、字重和文字颜色表达，不使用左侧蓝色竖线，避免线条感过强。
后台左侧导航共用规则：所有 merchant/operator 后台页面和后续新增页面必须复用共享导航脚本与共享样式，不允许在页面内自建 sidebar、topbar、menu 或写局部样式覆盖导航交互。新增菜单项只在对应 nav JS 中配置，并同步配置页面归属、顶部位置提示和最终选中项。
后台左侧导航结构规则：出现三级结构时必须采用"深色一级模块栏 + 浅色二三级菜单栏"的双栏布局，不把三级菜单塞回单一深色侧栏。一级模块栏只负责切换模块或直达无子级页面；展开态一级文案统一压缩为两个字短名（如 线路、订单、客户、设置），完整名称只用于 title、折叠提示和顶部位置提示；二级/三级菜单命名不超过5个字。
后台左侧导航交互规则：展开态下，一级模块鼠标停留和点击都可以切换右侧二级菜单；鼠标停留只临时预览，只有点击一级模块才固定停留在当前菜单，鼠标离开侧栏后回到已固定/当前页面所属菜单。折叠态只显示一级菜单图标，不显示一级文字、二级栏或三级菜单；折叠态鼠标停留一级图标时只快速显示一级模块名称提示，不展开右侧二级菜单、不改变侧栏宽度；只有点击一级图标才展开侧栏并展示菜单。
后台左侧导航视觉规则：一级模块栏宽度、二级栏宽度、折叠宽度统一由共享 CSS 变量控制，当前基准为一级栏约92px、二级栏约126px、折叠栏约64px；一级选中底色高度应与二级菜单项高度接近，不使用紫色作为后台导航主选中色。浅色二级栏顶部显示当前一级模块完整名称，标题文字起点必须与二级分组标题文字起点左对齐，字号和二级菜单保持一致，字重加粗（约 600）。二级分组展开箭头放在标题左侧；没有三级菜单的二级叶子项不显示箭头但必须预留箭头宽度，使文字起点和带箭头二级分组左对齐。三级菜单文字起点应比二级分组标题文字起点向右缩进约半个汉字。
后台左侧导航选中态规则：当前页面所在一级模块高亮、所在二级分组自动展开、最终叶子项保持选中；有子菜单时父级只负责展开。一级选中态可用蓝色整行底；二级/三级选中态只用浅灰底和普通文字色表达；所有二级/三级菜单文字保持 400 普通字重，不加黑、不加粗、不使用左侧蓝色竖线。
顶部多标签栏高度保持舒展但克制，标签高度约 32-34px；所有标签保持相同高度和字号，active 标签只改变颜色、边框或背景，不允许通过高度、字号、缩放造成放大缩小跳动。
左侧菜单折叠态必须仍可滚动；菜单项较多时图标列表不能被底部截断。折叠态子菜单、tooltip 等浮层如需露出侧栏外，应使用固定定位或其他不破坏 .nav-scroll 滚动的方式实现，不能通过关闭侧栏滚动来实现。
页面内容区的状态 Tab（如 全部/已上架/已下架/草稿/审核中、全部/合作中/暂停/终止 等）必须可以点击切换，不能只做静态展示；点击后切换 active 态，并筛选下方列表数据。
状态 Tab 使用 .tab-bar + .tab-item，并在 .tab-bar 上添加 data-filter-tabs；如果筛选目标不是紧邻表格，必须添加 data-filter-target 指向 tbody 或表格容器；每个 tab-item 应添加 data-filter-value，"全部"表示显示全部。
被筛选的表格行必须添加 data-tab-status，值可以包含一个或多个状态词，用空格分隔；例如 data-tab-status="销售中 即将售罄"。Tab 文案中的数量可以保留，但筛选值必须以 data-filter-value 为准，避免括号数量影响匹配。
表格操作列不允许出现无含义或无功能的独立箭头；如果有"更多"操作，必须显示"更多"文字并提供可点击下拉菜单，推荐使用 .dropdown.table-action-more + .table-more-toggle + .dropdown-menu.action-dropdown-menu，菜单中放具体操作项。更多按钮保持轻量文字样式，不使用明显描边按钮；下拉菜单使用小浮层、白底、轻边框、轻阴影，危险项使用 .dropdown-item.danger。只有确实能展开菜单时才显示下拉箭头；需要折叠的操作至少应有两个，若只有一个次要动作，直接在操作列显示该动作，不放入"更多"。

## 后台页面结构规则
一级列表页结构：标题区只包含页面标题和右侧操作按钮；不放面包屑、不放"共N个/共N条/总计N"等数据汇总、不放解释性副标题；页面内容区也不放大号 KPI 统计卡片、指标卡片横排或 `.kpi-grid/.kpi-card` 汇总区；即使页面需求写了"顶部统计卡片/右侧：共N条XX/共N个/核心指标"或类似汇总，也必须忽略大卡片汇总要求，数量只允许出现在状态Tab、分页、表格自身必要位置或轻量提示文字中；导航归属只通过顶部导航栏和左侧菜单选中态表达。列表列设计必须克制：非高频决策字段、资源/模板来源、字段来源、内部同步状态、产品内调整、辅助说明等不作为一级列表列展示，也不转移到详情、编辑或抽屉中作为被动说明展示；只有真实影响用户操作的业务数据、状态和动作才进入页面。交付前必须检查 .list-page-head 内不能出现"共"、"总计"、"合计"等汇总文案，并检查一级列表页内容区不能出现 `.kpi-grid` 或 `.kpi-card`。
一级列表页右侧操作按钮：主操作（如 新建）单独显示为 .btn-primary；导出、同步、AI智能上架和批量操作属于页面主操作，放在筛选工具条右侧或标题右侧；多个批量操作合并为"批量操作"下拉，推荐 .dropdown.table-action-more + .btn.btn-secondary.table-more-toggle + .dropdown-menu.action-dropdown-menu，按钮必须保留灰色边框次级按钮外观；禁止并列堆放"批量上架/批量下架/批量删除"等多个批量按钮。
一级列表页筛选区：使用 .filter-card 承载，但视觉上保持透明背景、无阴影、无额外边框；筛选项直接排列，输入框/选择器自身边框就是主要分隔；筛选项默认使用固定业务宽度，普通输入约220px、小下拉约136px、日期范围约318px，不为了填满整行而拉伸，条件较少时保持自然宽度。搜索/查询/重置按钮作为筛选条件流的最后一项，放在全部筛选项之后；不要放在第一个搜索框后，也不要和右侧主操作按钮混在一起。
一级列表页筛选区按钮：搜索/查询/重置使用 .btn-secondary 或同等级弱按钮；不要使用 .btn-primary、黑色按钮或大面积填充色。导出、同步、批量、新建/新增不放在筛选动作组里，应归入页面主操作区。
二级详情/新建/编辑页结构：标题区使用"返回 + 当前标题 + 右侧操作区"，默认不放面包屑；表单页返回或取消前提示未保存确认。
二级详情/新建/编辑页主操作唯一性：页面级主操作按钮组只能出现一次，统一放在标题区 .page-workbar 右侧；头部状态卡、摘要卡、详情卡只展示对象摘要、状态、关键指标或轻量上下文信息，不得再次放置与标题区相同的登记收款、申请退款、改价、提交、审批、更多等全局主操作按钮组。若页面需求同时描述"标题右侧操作"和"状态卡右侧操作"，以标题区操作为准，状态卡右侧应改为状态/指标/责任人/时间等非重复信息。
二级页面通用 UI 必须遵守 docs/secondary-page-ui-spec.md。标题区、返回文案、页面标题字号、Tab 宽度、Tab 下内容轨道、默认线条颜色、主色使用边界、详情页与编辑页差异均以该规范为准。
产品编辑类二级页面（如旅游线路编辑、研学线路编辑、自由行套餐编辑、单项服务编辑、外采方案包装、团期/航次/班期编辑等）必须遵守 docs/product-edit-page-spec.md。此类页面统一使用"顶部工作栏 + 摘要卡 + 步骤Tab + 整体表单卡 + 底部步骤按钮"结构，核心编辑内容保持约 1216px 最大宽度并左对齐；素材上传、多图、附件、明细卡、图标按钮、删除 hover 状态等按该 spec 落地。后续新增同类页面优先使用通用 edit-* 类，现有 route-blue-* 可作为当前旅游线路编辑页基准和迁移参考。
三级及复杂上下文页面：只有当层级、来源或业务归属无法通过返回按钮表达清楚时，才增加面包屑或简短归属信息，不做对象链说明区。

## CSS组件使用规范
状态标签：.tag .tag-blue(蓝) .tag-green(绿) .tag-orange(橙) .tag-red(红) .tag-gray(灰) .tag-purple(紫)
按钮：.btn .btn-primary(蓝) .btn-secondary(白边框) .btn-danger(红) .btn-warning(橙) .btn-success(绿)
KPI卡片：.kpi-card .blue/.green/.orange/.red/.purple（左侧彩色竖线），仅用于工作台、报表页、详情总览、分析看板等非一级列表场景；一级列表页禁止使用 KPI 卡做数据汇总。
弹窗/抽屉：普通弹窗使用 .modal-overlay > .modal .modal-sm(400px) .modal-md(600px) .modal-lg(720px)；表单/选择/维护类弹层默认使用右抽屉 .modal-overlay.drawer-overlay > .modal.drawer-modal，并保留右侧滑入/滑出的 transform 过渡；删除、上下架、复制、批量确认、未选择提示等打断确认使用居中 .modal.modal-sm.modal-confirm
搜索区：.filter-card 包裹；一级列表页使用 .filter-card.filter-card-compact，保持透明背景、无阴影、无额外边框；筛选项宽度使用 .filter-item(约220px) / .filter-item-sm(约136px) / .filter-item-wide(约318px) 三档，不使用 flex:1 拉伸铺满；日期范围用 .date-range，状态下拉可加 .filter-item-status
表格：.table-wrap > table
状态Tab筛选：.tab-bar[data-filter-tabs] + .tab-item[data-filter-value]；表格 tbody 行使用 data-tab-status 标注可筛选状态
表格更多操作：推荐 .dropdown.table-action-more + .table-more-toggle + .dropdown-menu.action-dropdown-menu + .dropdown-item；危险操作加 .dropdown-item.danger；禁止只放 ▼ 或 ▼更多 但无菜单；少于两个折叠动作时不使用更多菜单
